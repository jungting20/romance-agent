유효한 값으로 수정해도 validation 오류 2개가 계속 남는다

티켓
#17 · ticket-worker add 결과는 ready, fresh JSON 조회 시 외부 worker의 즉시 claim으로 in_progress
최소 재현 절차
  1. 세계관 수정 및 추가를 열고 기존 항목의 제목과 설명을 모두 비운다.
  2. 저장을 눌러 validation 오류 2개를 표시한다.
  3. 제목과 설명에 각각 비어 있지 않은 유효한 값을 입력한다.
  4. 다음 저장을 누르기 전에 alert, inline 오류와 각 field의 접근성 상태를 확인한다.
기대 결과
해결한 field의 오류만 즉시 사라지고 요약 개수가 2 → 1 → 0으로 갱신된다. 마지막 오류를 해결하면 validation alert, inline 오류, aria-invalid와 오류용 aria-describedby가 제거된다.
실제 결과
두 field 값이 모두 유효해도 “입력하지 않은 항목이 2개 있어요.” alert가 남고, 두 inline 오류와 두 field의 aria-invalid="true"가 그대로 유지된다. 같은 상태에서 다시 저장하면 PUT 200으로 성공하고 editor가 닫혀 표시 상태와 실제 유효성이 모순된다.
사용자 영향
사용자는 수정이 반영되지 않았다고 오인하며, 스크린리더 사용자는 유효한 field를 계속 invalid로 전달받는다. 저장을 재시도하면 진행할 수 있지만 validation 복구 과정의 신뢰성과 접근성이 훼손된다.
두 차례 clean-state 증거

실행 1: 오류 2개 발생 뒤 제목 비가 그친 온실 정상 저장, 설명 정상 저장 및 복구 경로를 확인하는 MSW 전용 임시 설명입니다.로 수정했다. 접근성 snapshot에 alert “2개”, 두 inline 오류, 두 invalid field가 모두 남았다.

실행 2: 페이지를 다시 불러 깨끗한 revision 1 상태에서 동일 절차를 반복하고 제목 복구된 제목, 설명 복구된 설명을 입력했다. DOM 평가 결과 alert “2개”, invalid field 두 개, 유효한 두 value가 동시에 확인됐다. 이어서 저장하자 PUT 200과 editor close가 확인됐다.

정상·오류·복구 결과

  • 초기 정상 상태에는 feedback alert가 없고 unchanged 상태의 저장은 disabled였다.
  • blank 제목·설명 제출은 오류 2개를 정확히 집계하고, 하나의 role="alert", field별 aria-invalid/aria-describedby, 첫 invalid 제목 focus를 제공했다.
  • 일회성 MSW 500은 retryable alert와 보존된 draft를 표시했다. dialog에서 Tab 두 번으로 다시 시도에 도달했고 Enter 재시도는 PUT 200, editor close, launch action focus, 저장 성공 status로 끝났다.
  • MSW 안에서 revision 1을 먼저 저장해 만든 409 conflict는 로컬 초안을 보존하고 최신 세계관 불러오기 action을 노출했다.
  • 일회성 MSW STORY_BIBLE_NOT_FOUND 404는 unavailable alert, 보존된 draft, disabled editor controls와 세계관 보기로 돌아가기 action을 노출했다.
  • network 기록에서 GET 200, 정상 PUT 200, 의도한 PUT 409/500/404를 확인했다.

중복된 관찰

1280px에서 Sheet가 384px로 표시되는 현상은 티켓 #16과 중복이다. conflict latest reload 또는 dirty close 확인에서 계속 편집 뒤 focus가 이탈하는 현상은 티켓 #15와 중복이다. 둘 다 이 보고서의 신규 티켓에 포함하지 않았다.

제외한 관찰

비동기 오류 응답 뒤 focus가 dialog 컨테이너에 놓였지만, 명시된 header close → feedback action Tab 순서로 두 번 안에 action에 도달하고 visible button focus가 확인되어 독립 결함으로 등록하지 않았다. Console의 favicon 404와 의도적으로 만든 409/500/404 resource 오류도 제품 흐름 결함에서 제외했다.

  • planner_setup_page: project 인자 없이 정확히 한 번 실행했다.

  • ticket-worker add: Medium 결함 문서 두 개로 티켓 #17을 등록했다. 명령 응답은 ready였으나 fresh 조회에서 자동 claim된 in_progress를 확인했다.
  • bash .codex/agents/tests/validate-bug-hunter.sh: 새 보고서 검사 전 기존 .codex/agents/bug-hunter.toml의 zellij-agent ticket-worker list --json 필수 문자열 assertion에서 종료 코드 1로 실패했다. 해당 기존 변경은 수정하지 않았다.

제약 및 미확인 항목

Playwright Test MCP allowlist에 viewport resize가 없어 요청된 375x800과 정확한 1280x800을 실행하지 못했다. 실제 검증은 1280x720이므로 모바일 및 800px 높이의 responsive 동등성은 미확인이다. 스크린샷을 보고서 asset 경로에 저장하려 했으나 MCP가 허용 root를 frontend/로 제한해 거부했으며, 소유 범위 밖 임시 파일을 만들지 않고 접근성 snapshot과 DOM 평가를 증거로 남겼다. 실제 backend, 외부 네트워크와 인증 상태는 확인하지 않았다. 이 보고서는 지정된 WorldEditorFeedback 흐름 밖의 품질을 주장하지 않는다.
