중복된 관찰

제외한 관찰

retry 성공 뒤 사라진 버튼 대신 document.body가 active element가 되는 동작은 승인 요구사항에 성공 후 포커스 복원 대상이 정해져 있지 않아 결함으로 등록하지 않았다. 또한 이미 로드된 페이지에서 테스트용 project ID로 SPA 전환하며 지연 GET을 적용했을 때 직전 query data가 보인 관찰은 지정된 최초 진입 흐름과 다른 합성 전환이고 clean initial-state 재현이 아니어서 제외했다.

수행한 검사

  • 정상 2회: /projects/silver-garden/write에서 h1 「은빛 정원의 약속」, 선택된 「원고 보기」 탭, 「자동 저장됨」 status, 문서 가로·세로 overflow 부재를 확인했다.
  • not-found 2회: /projects/no-such-project/write가 h1 「프로젝트를 찾을 수 없어요」와 「작품 서재로 돌아가기」 링크를 제공했다. 오류용 role=alert는 사용하지 않았고 h1으로 화면 목적을 전달했다.
  • not-found 키보드: 첫 Tab이 작품 서재 복귀 링크로 이동했고 transition 완료 뒤 3px focus ring이 표시됐다.
  • 일반 GET 실패 2회: MSW GET 500 자동 시도 2회 뒤 assertive alert 「작업 공간을 불러오지 못했어요. 잠시 후 다시 시도해 주세요.」와 「작업 공간 다시 불러오기」 버튼이 나타났다.
  • retry 2회: 첫 실행은 Tab과 Enter, 두 번째는 포인터 click으로 retry했다. 각 실행의 세 번째 GET이 200을 반환한 뒤 정상 workspace가 나타나고 오류 alert가 제거됐다.
  • URL normalization: tab=manuscript, unsupported string, boolean tab=true가 모두 replacement로 제거됐고 unrelated view=dense는 정상·not-found·일반 오류·retry 이후에도 보존됐다.
  • layout: 실제 1280x720에서 정상 workspace, 중앙 not-found, 중앙 오류 alert 모두 문서 scrollWidth=clientWidth, scrollHeight=clientHeight였다.
  • 네트워크: 2차 clean에서 not-found GET 404, 일반 오류 GET 500 두 번, retry GET 200, seed GET 200을 확인했다. 콘솔의 해당 404·500 resource 오류는 의도한 시나리오 응답과 일치했다.
  • 중복 검사: 탐색 전후 zellij-agent ticket-worker list --json으로 큐를 다시 조회했다.

제약 및 미확인 항목

Playwright-test MCP의 navigation과 click 도구가 대상 요청 완료까지 대기하고 runtime MSW handler는 새 document reload를 건너 유지되지 않아, clean 최초 진입 중의 짧은 skeleton과 sr-only 「작업 공간을 불러오는 중이에요.」 status를 직접 캡처하지 못했다. 요청된 375x800, 1280x800, 1440x900도 viewport resize 도구 부재로 미확인이다. 탐색 중 다른 작업의 커밋으로 공유 branch HEAD가 변경됐으나 이 agent는 제품 파일을 수정하지 않았다. 이 0건 결과는 두 번 확인한 정상·not-found·일반 오류·retry와 실제 1280x720에만 한정되며 애플리케이션 전체 무결함을 의미하지 않는다.
