탐색 범위

WorldEntryFields 컴포넌트에 한정해 세계관 편집기를 열고 기존·신규 row의 kind select, 제목, 설명, 신규 추가 포커스, validation 오류 연결, 저장 중 disabled/frozen, 키보드 Tab 순서, 긴 항목 스크롤을 확인했다. 정상 저장은 MSW의 PUT /api/projects/silver-garden/story-bible/world-entries 응답으로 검증했다.

제외 범위

제품 코드·테스트·설정·기존 문서 수정, 외부 네트워크와 실제 backend, 삭제·재정렬, 인물·원고·AI 도구, 애플리케이션 전체 감사는 제외했다. 인접한 WorldEditorSheet 폭과 폐기 확인 Dialog 포커스는 기존 티켓과 중복이므로 신규 등록하지 않았다.

중복된 관찰

데스크톱 Sheet가 384px로 표시되는 현상은 티켓 #16 「세계관 편집기 데스크톱 폭이 384px로 축소됨」과 중복이다. 폐기 확인에서 계속 편집 뒤 포커스가 document.body로 이탈하는 현상은 티켓 #15 「세계관 변경사항 폐기 취소 후 편집기 포커스 이탈」과 중복이다. 두 항목은 WorldEditorSheet 보고서에 이미 문서화되어 있어 신규 티켓을 만들지 않았다.

제외한 관찰

/favicon.ico 404 한 건은 이 컴포넌트의 사용자 흐름·요구사항과 무관한 정적 자원 관찰이라 제외했다.

수행한 검사

  • 정상 경로: 기존 row의 place → object 변경과 제목·설명 수정, 신규 row의 rule·제목·설명 입력, MSW PUT 200, 권위 응답에 따른 Sheet 닫힘과 성공 status를 확인했다.
  • 요청 계약: 기존 항목 ID는 updates에 보존되고 신규 항목은 ID 없이 additions에 포함되며, 응답이 신규 silver-garden-world-2 ID를 제공함을 확인했다.
  • 추가·포커스: 1차와 2차 clean-state 모두 신규 row 추가 직후 새 kind select가 active element가 됐다. 6개 row까지 늘릴 때마다 Sheet 내부 scrollTop이 증가해 새 select가 viewport 안으로 이동했다.
  • validation: 제목·설명 동시 공백은 오류 2개와 첫 제목 포커스, 제목만 입력한 2차 clean-state는 설명 오류 1개와 설명 포커스를 제공했다. 각 invalid control의 aria-invalid=true와 aria-describedby가 표시 메시지 ID에 정확히 연결됐다.
  • 키보드: 신규 row에서 Tab이 제목 → 설명 → 항목 추가 → 취소 → 저장 → 닫기 순서로 이동했고, 모달 아래 workspace에는 진입하지 않았다.
  • frozen: MSW PUT 응답을 2초 지연한 동안 status 「세계관을 저장하는 중이에요.」가 나타났고 모든 row 필드와 추가·닫기·취소·저장 버튼이 disabled였다.
  • 긴 항목: 6개 row와 긴 제목·설명을 입력해도 dialog의 scrollWidth는 clientWidth와 같았고, Sheet 내부 세로 스크롤로 마지막 설명과 footer에 접근 가능했다.
  • 콘솔·네트워크: 컴포넌트 동작 오류는 없었고 API 요청은 GET 200 두 건과 PUT 200 한 건이었다. 콘솔 오류는 범위 밖 favicon.ico 404뿐이었다.
  • 중복 검사: 탐색 전후 zellij-agent ticket-worker list --json으로 티켓 #1~#16을 대조했다.

제약 및 미확인 항목

허용된 Playwright-test MCP 도구 집합에는 viewport resize가 없고 window.resizeTo()도 headless viewport를 바꾸지 못해 375x800과 1280x800 반응형 결과는 확인하지 못했다. MSW의 기본 성공 경로와 브라우저 안에서 지연한 성공 응답으로 frozen을 검증했으며, 별도 실패·충돌 scenario 전환 수단이 제공되지 않아 request failure·revision conflict recovery는 미확인이다. 이 보고서는 지정 컴포넌트와 실제 실행 가능했던 경계에서만 0건을 뜻하며 애플리케이션 전체 무결함을 의미하지 않는다.
