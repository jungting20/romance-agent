# Narrative Memory 도메인 계약

## 목적과 책임

Narrative Memory는 Manuscript 장면에서 다시 만들 수 있는 청크별 장면 분석 결과와,
명시적 후속 워크플로에서 구성하는 추출 후보·분석 리비전·버전 JSON 스냅샷을 소유한다.
분석 결과나 후보를 작품의 사실로 취급하지 않으며 Manuscript나 Story Bible을
직접 변경하지 않는다.

## 보편 언어

- **장면 분석 리비전:** 특정 장면 본문 버전에 대해 수행한 분석의 버전
- **청크 분석:** 장면 본문의 유한한 부분과 그 부분에서 추출한 요약, 인물, 장소,
  관계 사건, 장소 사건을 묶은 미확정 결과
- **근거:** 청크 분석에서는 해당 청크 원문 기준 상대 문자 범위와 원문이며,
  스냅샷 후보에서는 장면·리비전·청크 출처를 함께 가진 근거
- **추출 후보:** 확인 전의 인물, 장소, 관계 사건, 장소 사건 해석
- **후보 상태:** `pending`, `approved`, `rejected`, `needs_review` 중 하나인 검토 상태
- **관계 사건:** 장면 순서와 근거를 가진 두 인물 사이의 시간적 사건
- **장소 사건:** 장면 순서와 근거를 가진 인물의 도착, 현재, 출발 사건
- **장면 관계 스냅샷:** 한 장면 리비전의 요약과 추출 후보를 담은 불변 JSON 구조
- **프로젝트 관계 스냅샷:** 현재 활성 장면 리비전들을 집계한 버전 있는 불변 JSON 구조

## 핵심 모델

- `ChunkExtraction`은 청크 요약과 미확정 인물, 장소, 관계 사건, 장소 사건을
  입력 순서대로 가진다. 각 근거는 청크 기준 상대 오프셋과 원문을 가진다.
- `AnalyzedChunk`는 청크 식별자, 숫자 순번, 장면 기준 시작·끝 위치, 불변 원문과
  `ChunkExtraction`을 묶는다.
- 스냅샷의 `Evidence`는 청크 식별자, 장면 식별자와 리비전, 시작과 끝 문자
  위치, 근거 원문을 가진다.
- `EntityCandidate`와 `PlaceCandidate`는 정규화 이름, 표시 이름, 별칭, 상태, 장면 리비전, 근거를 가진다.
- `RelationshipEventCandidate`와 `LocationEventCandidate`는 시간적 사건의 대상, 설명, 장면 순서,
  신뢰도, 상태, 근거를 가진다.
- `SceneRelationshipSnapshot`은 하나의 장면 리비전을, `ProjectRelationshipSnapshot`은 프로젝트의
  현재 활성 장면 리비전과 추출 후보를 불변 튜플로 보관한다.

## 불변 조건

- 모든 추출 결과는 미확정 해석이며, 사용자 확인 없이 자동으로 사실이 되어서는 안 된다.
- 명시적 장면 분석의 호출자는 불변 장면 본문과 필요할 때만 알려진 인물·장소 정체성 카탈로그를
  함께 제공한다. Narrative Memory는 이 입력을 얻기 위해 다른 도메인을 조회하지 않는다.
- 장면 분석은 장면 본문을 최대 300자, 50자 중첩 청크로 나누고 숫자 순서대로 직렬
  처리하며 각 청크를 모델에 한 번만 전달해야 한다.
- 성공한 장면 분석은 청크별 구조화 추출물과 청크 기준 상대 근거를 입력 순서대로
  반환한다. 어느 청크든 실패하면 뒤 청크를 호출하지 않고 부분 결과를 반환하지 않는다.
- 청크별 구조화 추출물은 작품의 사실, 영속 후보, 장면 스냅샷이 아니다. 새 영속 식별자나
  후보 상태를 생성·할당하지 않으며, 근거를 장면 절대 위치로 변환하거나 청크 간 병합·중복
  제거를 하지 않는다.
- 관계와 장소 사건은 호출자가 정확히 제공한 알려진 정체성 키 또는 해당 청크 추출물
  안의 로컬 참조를 사용할 수 있다. 장면 분석은 이 참조를 영속 도메인 정체성으로 해석하지 않는다.
- 모델 제공자와 프롬프트 파일은 인프라스트럭처 관심사다. 장면 분석은 다른 도메인의
  상태를 읽거나 변경하지 않으며 추출물을 스냅샷에 자동 저장·병합·승인하지 않는다.
- 명시적 후속 워크플로가 새 추출물을 장면 또는 프로젝트 스냅샷 후보로 채택할 때는 새
  후보를 `pending`으로 취급하며, 기존 안정 후보와 근거가 일치하는 명시적 병합 규칙만 기존
  승인 상태를 보존할 수 있다.
- 모든 모음 필드는 튜플이어야 하며 스냅샷은 생성 후 변경할 수 없어야 한다.
- 빈 프로젝트 관계 스냅샷은 스냅샷 버전 `0`과 빈 모음을 가져야 한다.
- 재분석은 변경된 장면의 `pending` 후보를 해당 장면의 새 리비전 후보로 대체해야 한다.
- 재분석 후 근거가 사라진 확인 사실은 `needs_review`로 전환해야 하며 재확인 전에는 확정 사실로
  소비되어서는 안 된다.
- 관계와 장소는 정적 속성이 아니라 장면 순서에 따른 시간적 관계 사건과 장소 사건으로 표현해야 한다.
- 미해결 인물과 장소의 정체성은 장면별 복제본이 아니라 프로젝트 범위의 정규화 이름과 별칭으로
  해석해야 한다. 여러 장면의 근거는 하나의 정체성에 합치되 각 근거의 장면과 리비전 출처를 보존해야
  한다.
- 재분석 군집에 `approved` 또는 `needs_review` 정체성이 하나 있으면 그 식별자를 보존해야 하며, 서로
  다른 안정 식별자들이 하나의 미해결 정체성으로 합쳐지는 충돌은 거부해야 한다. 관계 사건과 장소
  사건의 안정 식별자도 재분석으로 교체해서는 안 된다. 안정 정체성이 없는 군집의 대표는 입력 순서와
  무관한 결정적 규칙으로 선택해야 한다.
- 프로젝트 인물·장소 후보 식별자는 두 목록 전체에서 유일해야 하고, 관계·장소 사건 식별자도 두 사건
  목록 전체에서 유일해야 한다. 프로젝트 범위에서 같은 정규화 이름 또는 별칭으로 연결되는 정체성은
  하나의 후보로 합쳐져야 한다.
- 청크 분석의 각 근거는 해당 청크 원문 기준 범위 안에 있어야 하며 근거 원문은 불변
  청크 원문의 동일 상대 범위와 정확히 일치해야 한다. 반환 순서는 숫자 청크 순번을 따른다.
- 청크 식별자는 장면 식별자, 장면 리비전, 0부터 시작하는 연속 숫자 순번을 정확히 인코딩해야 한다.
  순번 `n`의 시작 위치는 `n * 250`이고 청크 너비는 `1..300`이어야 하며, 마지막 청크를 제외한 모든
  청크의 너비는 `300`이어야 한다. 인접 청크의 50자 중첩 범위 원문은 정확히 같아야 하며, 여러 청크가
  있으면 마지막 청크는 중첩 범위 너머의 문자를 하나 이상 포함해야 한다.
- 장면 스냅샷과 프로젝트 스냅샷은 각각 지원하는 스키마 버전만 허용하며 서로 호환되어야
  한다. 혼합되거나 알 수 없는 버전은 공개 병합 및 디코딩 경계에서 거부해야 한다.
- 스냅샷 관계 사건과 장소 사건의 장면 순서는 포함하는 장면 순서와 같아야 하고, 신뢰도는 유한한
  `0.0..1.0` 닫힌 구간이어야 한다.
- 프로젝트 병합 전 모든 후보의 장면·리비전 출처, 활성 장면 리비전의 유일성, 근거 범위, 사건 시간
  순서, 스키마 호환성을 다시 검증해야 한다. 관계 사건의 주체와 객체는 프로젝트 인물 목록을,
  장소 사건의 인물과 장소는 각각 프로젝트 인물 및 장소 목록을 참조해야 한다. 프로젝트 스냅샷의
  사건 참조는 정체성 합치 후 보존된 인물·장소 후보 식별자로 정규화해야 한다. 참조 정규화로 같은
  사건 정체성이 된 시간적 후보는 근거 중첩을 다시 적용해 합치고, 서로 떨어진 발생은 유지하며, 서로
  다른 안정 사건 식별자가 하나로 합쳐지는 충돌은 거부해야 한다.
- `approved` 관계 사건은 두 인물 후보가 모두 `approved`인 경우에만 허용하며, `approved` 장소 사건은
  인물과 장소 후보가 모두 `approved`인 경우에만 허용한다. 미해결 후보를 참조하는 `pending` 사건은
  허용하되 병합 결과에서 승인 의존성이 해소되지 않은 승인 사건은 `needs_review`로 전환해야 한다.
- 활성 리비전보다 오래된 근거는 `needs_review` 후보에만 남을 수 있다. 병합은 오래된 근거를 가진
  `approved` 후보와 사건을 `needs_review`로 전환해야 하며, 병합을 거치지 않고 저장하거나 디코딩한
  스냅샷에서 이 조건을 위반하면 거부해야 한다.
- 프로젝트 스냅샷 의미 검증은 하나의 도메인 검증기가 소유한다. 병합, JSON 인코딩·디코딩, 저장소
  쓰기와 읽기 경계는 동일한 스키마, 유일성, 출처, 근거, 참조, 승인 의존성 불변 조건을 적용해야 한다.
- 재분석 근거가 기존 승인 후보의 모든 실질적 근거를 포함하면 순서나 추가 근거와 관계없이 승인
  상태와 안정 식별자를 유지하고 새 리비전 메타데이터 및 전체 새 근거를 채택해야 한다. 기존 근거가
  하나라도 빠지면 `needs_review`로 전환해야 한다.

## 유스케이스

### 장면 분석

호출자가 Manuscript에서 전달한 특정 장면 리비전의 불변 본문과 선택적 알려진 인물·장소 정체성
카탈로그를 입력한다. 본문을 300자/50자 중첩 청크로 나누어 숫자 순서대로 직렬 분석하고 각 청크를
모델에 한 번만 전달한다. 성공하면 청크별 구조화 추출물과 청크 기준 상대 근거를 순서대로 반환한다.
추출물은 사실, 영속 후보, 장면 스냅샷이 아니며 자동 저장·병합·승인하지 않는다. 어느 청크든 실패하면
부분 결과를 반환하지 않는다.

### 장면 재분석

장면 리비전이 바뀌면 새 청크별 분석 결과를 받은 명시적 후속 애플리케이션 유스케이스가 이전
`pending` 후보를 해당 장면의 새 리비전 후보로 대체하고, 기존 확인 사실의 근거가 사라졌다면
`needs_review`로 전환한다. 단순 장면 분석은 이 변경을 수행하지 않는다.

### 프로젝트 스냅샷 구성

장면 순서와 활성 리비전에 따라 장면 스냅샷을 집계해 새 버전의 프로젝트 관계 스냅샷을 만든다.

### 후보 검토와 사실 확인

후보와 근거를 검토 흐름에 제공한다. 사용자가 확인하면 명시적 애플리케이션 유스케이스가 Story Bible에
확인된 관계 사건 또는 장소 사건 사실을 등록하도록 조정한다. Narrative Memory는 Story Bible을 직접
변경하지 않는다.

## 입력과 출력

- Manuscript의 프로젝트 식별자, 장면 식별자, 장면 순서, 장면 리비전, 장면 본문을 명시적
  애플리케이션 유스케이스를 통해 입력받는다.
- 장면 분석은 근거가 포함된 미확정 인물, 장소, 관계 사건, 장소 사건과 요약을 청크별로
  반환한다.
- 별도의 명시적 스냅샷 유스케이스는 불변 장면 관계 스냅샷과 버전이 있는 프로젝트 관계 JSON
  스냅샷을 반환한다.
- Writing Assistant의 명시적 일관성 검사 유스케이스에 필요한 범위로 제한한 장면 요약을 제공한다.

## 책임지지 않는 영역

- Manuscript 장면, 장면 순서, 장면 리비전, 원고 본문의 소유와 변경
- Story Bible 사실의 직접 생성, 변경, 삭제
- 후보의 자동 승인 또는 자동 사실 전환
- Writing Assistant 제안 또는 일관성 진단 생성
- HTTP API, 백그라운드 큐, 저장소, 데이터베이스, 외부 모델 및 제공자의 선택
