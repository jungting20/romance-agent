# 집필 에디터 스크롤 경계 설계

## 문제

집필 워크스페이스에서 원고가 세로로 길어지면 중앙 에디터 영역이 아니라 브라우저 문서 전체에 스크롤이 생길 수 있다. 현재 중앙 `main`에는 `overflow-y-auto`가 있지만, 화면 최상위 컨테이너가 `min-h-svh`만 사용하므로 자식 콘텐츠가 커질 때 화면 자체도 함께 늘어날 수 있다.

## 목표

- 집필 워크스페이스의 헤더와 도구 내비게이션은 뷰포트 안에 유지한다.
- 장 제목, 메타데이터, 원고 입력란을 포함한 중앙 에디터 영역만 세로로 스크롤한다.
- 긴 원고에서도 브라우저 문서 자체에는 세로 스크롤이 생기지 않는다.
- 데스크톱의 고정·리사이즈 패널 구성과 모바일의 시트 구성을 모두 보존한다.

## 선택한 접근

집필 워크스페이스 최상위 컨테이너를 `100svh` 높이로 제한하고, 헤더 아래의 flex 경로에 `min-h-0`과 필요한 `overflow-hidden`을 유지한다. 중앙 에디터 `main`은 제한된 남은 높이를 받아 유일한 원고 세로 스크롤 컨테이너가 된다.

대안으로 전역 `body`의 오버플로를 막을 수 있지만 다른 라우트까지 영향을 줄 수 있어 제외한다. `textarea` 자체에만 스크롤을 맡기면 장 제목과 편집 용지가 고정되어 에디터 영역 전체가 스크롤되어야 한다는 요구를 충족하지 못하므로 제외한다.

## 변경 범위

- `frontend/src/pages/writing-workspace/writing-workspace-page.tsx`
  - 로딩 화면과 정상 집필 화면의 높이 경계를 뷰포트에 고정한다.
  - 모바일, 고정 사이드바, 리사이즈 패널 분기에서 중앙 에디터의 높이 축소와 오버플로 경계를 명시한다.
- `frontend/src/pages/writing-workspace/writing-workspace-page.test.tsx`
  - 집필 화면이 고정된 뷰포트 경계와 중앙 편집 스크롤 소유권을 렌더링하는 회귀 검증을 추가한다.

`Manuscript`의 모델, 저장, 자동 저장, 선택 범위, API 계약은 변경하지 않는다. 도메인 의미가 바뀌지 않으므로 `docs/domains/manuscript.md` 수정은 필요하지 않다.

## 수용 기준

1. 긴 원고를 연 집필 화면에서 `document.documentElement.scrollHeight`가 뷰포트 높이를 넘어 문서 스크롤을 만들지 않는다.
2. 중앙 에디터 영역은 콘텐츠가 사용 가능한 높이를 넘으면 자체 세로 스크롤을 제공한다.
3. 중앙 에디터를 스크롤해도 상단 헤더와 왼쪽 도구 내비게이션의 화면 위치는 유지된다.
4. 데스크톱 고정 사이드바, 데스크톱 리사이즈 패널, 모바일 시트 구성이 기존과 동일하게 동작한다.
5. 원고 입력, 선택 범위 추적, 자동 저장 및 보조 패널 동작에 회귀가 없다.

## 검증

- 관련 페이지 테스트를 먼저 실패시키고 최소 레이아웃 변경 후 통과시킨다.
- 실제 브라우저에서 작은 높이와 데스크톱·모바일 너비로 문서 스크롤 부재와 중앙 에디터 스크롤을 확인한다.
- 프론트엔드 전체 검사와 빌드를 실행한다.
