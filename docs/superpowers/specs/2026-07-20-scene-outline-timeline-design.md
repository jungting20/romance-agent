# 장면 구성 타임라인 설계

## 배경

Muse의 현재 핵심 흐름은 프로젝트 생성, 장면 집필, 원고 자동 저장, 스토리 바이블 조회와 수정이다.
Manuscript는 장면 배열과 활성 장면을 이미 소유하며, 별도의 진행 중 티켓인 `원고 장면 추가 및 목차
전환`이 장면 추가와 선택을 현재 집필 작업공간에 연결하도록 설계되어 있다.

작가가 단편을 실제로 완성하려면 장면을 추가하는 것만으로는 부족하다. 전체 이야기 순서를 보면서 각
장면의 목적과 진행 상태를 계획하고, 순서를 바꾸고, 필요 없는 장면을 제거한 뒤 곧바로 해당 원고를
집필할 수 있어야 한다. AI 기반 구조 생성은 뒤로 미루고 기본 집필 기능에 집중한다.

## 목표

집필 작업공간에 순서형 장면 구성 타임라인을 추가한다. 사용자는 10~30개 장면 규모의 단편을 한
화면에서 계획하고, 순서를 조정하고, 각 장면으로 이동해 집필하며, 기존 자동 저장과 충돌 해결 흐름을
통해 구성과 원고를 보존할 수 있어야 한다.

초기 설계는 단편에 맞게 가볍게 유지하되 장면 수가 늘어나도 같은 순서형 모델을 사용할 수 있게 해
향후 장편 확장을 막지 않는다.

## 선행 조건

이 기능은 다음 승인 문서에 정의된 장면 추가와 선택 동작을 선행 기반으로 사용한다.

- `docs/superpowers/specs/2026-07-16-manuscript-scene-add-design.md`
- `docs/superpowers/plans/2026-07-16-manuscript-scene-add.md`

이번 기능은 그 티켓의 장면 추가, 활성 장면 선택, 새 장면 포커스, 구조 충돌 기반을 중복 구현하지
않는다. 구현을 시작하기 전에 선행 동작이 현재 기준 브랜치에 통합되었는지 확인해야 한다.

## 범위

### 포함

- 집필 작업공간의 `구성`과 `집필` 보기 전환
- 이야기 순서대로 배치한 세로형 장면 카드 타임라인
- 장면 제목, 한 줄 장면 계획, 수동 초고 상태 표시와 수정
- 드래그와 명시적 위·아래 이동을 통한 장면 재정렬
- 확인 절차를 거친 장면 삭제와 활성 장면 재선택
- 타임라인 카드에서 해당 장면의 집필 보기로 이동
- 기존 전체 원고 자동 저장, 실패 재시도, 리비전 충돌 해결과의 통합
- 새 장면 필드가 없는 기존 원고의 호환 읽기
- 관련 Manuscript 도메인 계약, OpenAPI 계약, 소비자 타입, 목과 저장 표현의 동기화
- 도메인, 화면 통합, 저장·충돌, 호환성, 접근성 및 E2E 검증

### 제외

- 부, 장, 화 또는 막을 별도 컨테이너로 표현하는 계층형 목차
- 3막 구조나 로맨스 비트 템플릿
- AI 장면 생성, AI 구조 추천 또는 자동 상태 변경
- 공동 편집, 댓글, 사용자 계정과 권한
- 원고 버전 이력과 임의 시점 복원
- DOCX, PDF 또는 다른 원고 형식 내보내기
- 새로운 장면 전용 저장 API

## 사용자 경험

### 보기 전환

집필 작업공간 상단에 `구성 / 집필` 전환을 둔다. 구성 보기는 같은 프로젝트 작업공간 안에서
열리며, 다시 집필 보기로 돌아왔을 때 마지막 활성 장면을 유지한다. 보기 상태는 작업공간 URL 상태로
표현하여 새로고침과 브라우저 뒤로 가기에도 현재 보기가 일관되게 복원되게 한다.

### 순서형 타임라인

구성 보기는 장면 배열의 이야기 순서를 위에서 아래로 표시한다. 각 카드에는 다음 정보와 조작이 있다.

- 현재 표시 번호
- 장면 제목
- 한 줄 장면 계획
- `계획`, `집필 중`, `초고 완료` 상태
- 순서 변경 손잡이
- 편집 및 삭제 메뉴

카드 본문을 선택하면 그 장면을 활성화하고 집필 보기로 이동한다. 제목, 계획과 상태 수정은 카드의
편집 동작으로 여는 작은 시트에서 처리한다. 구성 화면은 원고 본문 편집기를 동시에 표시하지 않는다.

### 장면 추가

선행 기능의 장면 추가 동작을 구성 보기에서도 사용한다. 새 장면은 현재 선택한 장면 바로 다음에
추가되고 활성 장면이 된다. 새 장면의 제목과 본문 기본값은 선행 설계를 따르며, 장면 계획은 빈
문자열, 초고 상태는 `planned`로 시작한다.

### 장면 재정렬

사용자는 손잡이로 카드를 드래그하여 이야기 순서를 바꿀 수 있다. 같은 기능을 키보드와 터치 환경에서
사용할 수 있도록 각 카드 메뉴에 `위로 이동`과 `아래로 이동`을 제공한다. 첫 카드의 위로 이동과 마지막
카드의 아래로 이동은 사용할 수 없다.

재정렬이 완료되면 카드 순서, 표시 번호, 집필 보기의 장면 번호와 작업공간 헤더가 같은 결과를
보여야 한다. 재정렬 자체는 장면 식별자, 본문, 계획, 상태 또는 Story Bible 문맥 참조를 변경하지
않는다.

### 장면 삭제

삭제 전에 장면 제목과 원고 본문도 함께 제거된다는 사실을 확인한다. 마지막 남은 장면은 삭제할 수
없다. 활성 장면을 삭제하면 삭제된 위치의 다음 장면을 활성화하고, 다음 장면이 없으면 이전 장면을
활성화한다. 활성 장면이 아닌 장면을 삭제하면 현재 활성 장면을 유지한다.

## 도메인 모델과 불변 조건

별도 Outline 도메인은 만들지 않는다. 장면 계획과 초고 상태는 원고 장면의 집필 생명주기에 속하므로
Manuscript가 소유한다.

기존 장면에 다음 값을 추가한다.

```text
outline: string
draftStatus: planned | drafting | draft_complete
```

`outline`은 비어 있을 수 있다. 이 기능은 제목과 계획에 임의의 새 최대 길이 제한을 도입하지 않는다.
`draftStatus`는 지원하는 세 값 중 하나여야 하며 사용자의 명시적 변경으로만 바뀐다. 본문 입력이나
저장 성공이 상태를 자동 변경해서는 안 된다.

Manuscript는 다음 상태 전이를 소유한다.

- 존재하는 장면의 제목, 계획과 초고 상태 갱신
- 존재하는 장면을 지정한 유효 위치로 이동
- 마지막 장면이 아닌 존재하는 장면 삭제
- 삭제 뒤 유효한 활성 장면 결정

장면 배열 순서가 이야기 순서의 기준이다. 현재 모델의 장면 번호는 배열 순서에 따라 1부터 연속으로
다시 계산한다. 장면을 이동하거나 삭제해도 남은 장면의 식별자는 유지한다. 모든 연산은 입력 원고와
기존 장면을 직접 변경하지 않고 새 원고를 반환해야 한다.

존재하지 않는 장면 수정·이동·삭제, 범위를 벗어난 이동 위치, 마지막 장면 삭제는 전체 요청을
거부하며 부분 결과를 반환하지 않는다.

## 데이터 흐름과 저장 계약

### 일반 변경

1. 사용자가 카드 편집, 재정렬 또는 삭제를 요청한다.
2. 작업공간 기능이 Manuscript의 순수 도메인 연산을 호출한다.
3. 성공한 새 원고를 기존 자동 저장의 로컬 초안으로 채택한다.
4. 타임라인, 집필 보기와 헤더가 같은 활성 장면과 순서를 다시 렌더링한다.
5. 기존 자동 저장 지연 뒤 전체 원고와 기대 리비전을 `saveManuscript`에 전달한다.
6. 저장 성공 시 서버 원고와 새 리비전을 승인된 기준 상태로 채택한다.

장면 구성만을 위한 새 API 작업은 추가하지 않는다. 장면 제목, 계획, 초고 상태와 순서는 전체 원고의
일부로 저장한다.

### API와 표현 동기화

기존 원고 장면을 포함하는 OpenAPI 스키마에 `outline`과 `draftStatus`를 반영한다. 같은 승인 기준으로
프런트엔드 계약 타입, API 어댑터, MSW 요청·응답과 목 데이터, 그리고 영향을 받는 백엔드 운송 또는
저장 표현을 동기화한다. OpenAPI 변경은 저장소의 OpenAPI 전담 에이전트만 작성하고 Main이 정확한
기준을 승인한 뒤 구현에 사용한다.

현재 승인된 전체 원고 저장의 기대 리비전과 응답 리비전 의미는 변경하지 않는다. 경로, 메서드,
operationId, 기존 오류 코드 또는 상태 코드도 이 기능 때문에 새로 만들거나 바꾸지 않는다.

### 기존 데이터 호환성

새 필드가 없는 기존 장면은 읽기 경계에서만 다음 기본값으로 해석한다.

- `outline`: 빈 문자열
- 빈 본문: `planned`
- 비어 있지 않은 본문: `drafting`

기존 데이터에 대한 상태 추론은 최초 호환 읽기에만 적용한다. 도메인 상태가 만들어진 뒤에는 원고
본문 변화로 초고 상태를 다시 계산하지 않는다. 다음 정상 저장은 명시적인 새 필드를 포함한다. 별도의
일괄 데이터 마이그레이션이나 시작 시 전체 재기록은 하지 않는다.

## 저장 실패와 리비전 충돌

일반 저장 실패 시 현재 로컬 원고, 장면 구성과 활성 장면을 유지한다. 기존 저장 상태 표시와 재시도
동작으로 같은 최신 초안을 다시 저장한다.

다른 탭이나 위치에서 서버 원고가 먼저 저장되어 리비전 충돌이 발생했을 때 장면 편집, 재정렬과 삭제를
자동 병합하지 않는다. 충돌 화면은 현재 로컬 초안을 보존한 채 다음 두 선택을 제공한다.

- **내 구성 유지:** 현재 로컬 원고 전체를 서버의 최신 리비전에 대해 다시 저장한다. 이 선택이 서버의
  최신 원고 전체를 로컬 버전으로 대체할 수 있음을 명확히 알린다.
- **서버 최신본 적용:** 로컬의 미저장 구성과 원고를 버리고 서버 원고와 리비전을 채택한다.

재시도나 서버 최신본 조회가 실패하면 로컬 초안을 유지하고 다시 시도할 수 있게 한다. 사용자가 어느
선택도 하기 전에는 충돌 상태에서 추가 구조 변경을 허용하지 않는다.

## 접근성 및 반응형 동작

- 구성과 집필 전환, 카드 본문, 편집, 삭제와 순서 변경에 명시적인 접근 가능한 이름을 제공한다.
- 현재 활성 장면은 색상 외에 `aria-current` 또는 동등한 상태로 전달한다.
- 드래그는 유일한 재정렬 수단이 아니며 위·아래 이동을 키보드로 사용할 수 있어야 한다.
- 재정렬, 삭제, 상태 변경과 저장 결과를 비방해성 live region 또는 기존 상태 패턴으로 알린다.
- 삭제와 리비전 충돌은 명시적인 제목, 영향 설명, 취소 가능한 동작을 제공한다.
- 모바일에서는 세로 타임라인을 한 열로 유지하고 카드 조작을 메뉴로 사용할 수 있게 한다.

## 테스트 전략

### 도메인 단위 테스트

- 장면 제목, 계획과 지원 초고 상태 갱신
- 지원하지 않는 상태와 존재하지 않는 장면 거부
- 배열 순서 변경과 1부터 연속인 표시 번호 재계산
- 재정렬 뒤 식별자, 본문, 계획, 상태와 문맥 참조 보존
- 활성 및 비활성 장면 삭제 뒤 활성 장면 결정
- 마지막 장면과 존재하지 않는 장면 삭제 거부
- 모든 성공과 실패 경로에서 입력 원고 불변성

### 화면 및 통합 테스트

- 구성과 집필 보기 전환 및 URL 상태 복원
- 순서대로 표시한 카드의 제목, 계획, 상태와 활성 표시
- 카드 편집 뒤 자동 저장 초안과 UI 갱신
- 드래그 및 위·아래 이동의 동일한 재정렬 결과
- 카드 선택 뒤 해당 장면 집필 보기와 헤더 갱신
- 삭제 확인, 취소, 활성 장면 대체와 마지막 장면 보호
- 모바일 한 열 배치, 키보드 조작과 상태 안내

### 저장, 계약과 호환성 테스트

- 저장 요청에 모든 장면의 새 필드와 재정렬된 순서가 포함됨
- 저장 실패 뒤 로컬 구성 유지와 재시도
- 리비전 충돌에서 내 구성 유지와 서버 최신본 적용
- 충돌 해결 실패 뒤 로컬 초안 보존
- 새 필드가 없는 빈 본문과 기존 본문의 호환 기본값
- 호환 읽기 뒤 새 필드를 포함한 정상 재저장
- 승인된 OpenAPI, 프런트엔드 타입·어댑터·MSW와 영향을 받는 백엔드 표현의 일치

### E2E

프런트엔드 구현과 읽기 전용 애플리케이션 리뷰가 끝난 뒤 프로젝트의 E2E 절차를 따른다. 최소 흐름은
장면 계획 작성, 장면 추가, 순서 변경, 장면 집필, 자동 저장과 재접속 복원을 포함한다. 저장 실패와
충돌 해결의 사용자 복구 경로도 검증한다.

## 완료 기준

- 사용자가 10~30개 장면을 이야기 순서대로 보고 계획할 수 있다.
- 제목, 한 줄 계획과 초고 상태를 편집해도 원고 본문과 Story Bible 문맥 참조가 보존된다.
- 드래그, 키보드와 터치 환경에서 같은 장면 순서 변경 결과를 얻는다.
- 카드에서 해당 원고로 이동했을 때 타임라인, 헤더와 편집기가 같은 활성 장면을 가리킨다.
- 장면 삭제가 명시적으로 확인되며 마지막 장면과 유효한 활성 장면 불변 조건이 보존된다.
- 구성 변경이 자동 저장, 실패 재시도와 두 가지 충돌 해결 경로를 거쳐 보존되거나 명시적으로
  폐기된다.
- 새 필드가 없는 기존 원고를 읽고 데이터 손실 없이 새 표현으로 저장할 수 있다.
- 새 API 작업이나 AI, 계층형 목차, 템플릿, 버전 이력 또는 내보내기 기능이 도입되지 않는다.
- 구현, OpenAPI 계약, 테스트와 `docs/domains/manuscript.md`가 같은 책임과 규칙을 설명한다.

## 문서 및 전달 경계

이 기능은 Manuscript의 모델, 불변 조건과 유스케이스를 변경하므로 구현과 같은 변경에서
`docs/domains/manuscript.md`를 갱신한다. 도메인 간 의존 방향은 바뀌지 않으므로
`docs/domains/README.md`의 컨텍스트 맵 변경은 필요하지 않다.

소비자 계약의 장면 표현이 바뀌므로 `docs/api/openapi.yaml` 변경과 Main의 새 기준 승인이 필요하다.
구현은 승인된 같은 기준을 사용하는 프런트엔드와 영향을 받는 백엔드 경계로 나누고, 각 애플리케이션의
전체 검증과 저장소 지침에 정의된 읽기 전용 리뷰 및 E2E 순서를 따른다.

## 승인 기록

사용자는 2026-07-20에 AI를 제외하고 단편부터 시작해 장편으로 확장 가능한 기본 집필 기능에 집중하는
방향을 선택했다. 세 가지 구조 접근 중 순서형 장면 타임라인을 선택했고, 기능 범위와 흐름, Manuscript
책임 경계와 선행 티켓 관계, 화면 동작과 충돌 처리, 호환성·테스트·완료 기준을 각각 승인했다.
