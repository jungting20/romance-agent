# 원고 장면 추가 설계

## 목표

집필 워크스페이스의 기존 `새 장면 추가` 버튼을 활성화한다. 사용자는 별도 입력 단계 없이
새 장면을 만들고 곧바로 집필을 시작할 수 있어야 한다. 추가된 장면은 원고 목차에 나타나고
활성 장면이 되며, 기존 전체 원고 자동 저장과 리비전 충돌 흐름을 통해 보존된다.

## 현재 상태

- Manuscript는 장면 배열과 `activeSceneId`를 이미 소유한다.
- `SceneTree`는 장면을 표시하지만 추가 버튼이 비활성화되어 있고 장면 선택도 동작하지 않는다.
- 집필 워크스페이스는 활성 장면을 편집하고 전체 Manuscript를 기존
  `PUT /manuscripts/{manuscriptId}` 작업으로 자동 저장한다.
- API 스키마와 저장 작업은 이미 복수 장면을 지원하므로 새 엔드포인트는 필요하지 않다.

## 범위

### 포함

- 장면을 즉시 추가하는 Manuscript 도메인 연산
- 기존 장면을 활성화하는 Manuscript 도메인 연산
- 원고 목차의 장면 추가, 선택, 활성 상태 표시
- 새 장면으로 전환한 편집기와 워크스페이스 헤더
- 기존 자동 저장, 재시도, 리비전 충돌 흐름에서 장면 구조 변경 보존
- 데스크톱과 모바일의 포커스 및 목차 전환 동작
- 관련 도메인 계약, 단위 테스트, 화면 통합 테스트와 E2E 테스트

### 제외

- 장면 이름 변경
- 장면 순서 변경
- 장면 삭제
- 부 또는 중첩 목차 계층
- 새 API 엔드포인트

## 도메인 설계

Manuscript 도메인은 `addScene`과 `selectScene`에 해당하는 순수 연산을 제공한다. 두 연산은
입력 원고를 변경하지 않고 새 원고를 반환한다.

`addScene`은 애플리케이션 계층에서 생성된 비어 있지 않은 장면 식별자를 받는다. 도메인은
동일한 식별자가 이미 존재하면 요청을 거부한다. 새 장면은 다음 값으로 생성한다.

- 제목: `제목 없는 장면`
- 장 번호: 기존 장면의 가장 큰 장 번호 + 1
- 본문: 빈 문자열
- 관련 인물 및 세계관 참조: 빈 배열

새 장면은 장면 배열 끝에 추가되며 동시에 `activeSceneId`가 된다. ID 생성은 브라우저 API에
의존하므로 순수 도메인 밖의 집필 워크스페이스 계층이 담당한다.

`selectScene`은 존재하는 장면 식별자만 받을 수 있다. 존재하지 않는 장면을 선택하면 요청을
거부하며, 성공하면 장면 내용이나 순서를 바꾸지 않고 `activeSceneId`만 변경한다.

`docs/domains/manuscript.md`에는 장면 추가와 활성 장면 선택 유스케이스, 중복 ID 거부,
다음 장 번호 결정, 불변 갱신 규칙을 같은 변경에서 반영한다.

## UI 구조와 상호작용

`SceneTree`는 Manuscript와 함께 다음 명시적 입력을 받는다.

- 장면 추가 콜백
- 장면 선택 콜백
- 추가 비활성 상태

기존 `새 장면 추가` 아이콘 버튼을 활성화한다. 한 번 활성화할 때 정확히 한 장면을 만들며,
별도 대화상자나 인라인 제목 입력은 표시하지 않는다. 목차의 모든 장면 버튼은 선택 가능해야
하며 활성 장면은 시각적 강조와 `aria-current`로 구분한다.

장면을 추가하거나 선택하면 집필 워크스페이스가 도메인 연산으로 자동 저장 초안을 변경하고
본문 선택 범위를 초기화한다. 편집기, 편집기 제목과 장 번호, 워크스페이스 헤더는 새
`activeSceneId`에서 파생한다. 현재 헤더의 고정된 `제1장` 표시는 활성 장면 번호를 사용하도록
교정한다.

추가 직후 `원고 본문` 입력란으로 포커스를 이동하고 `N장 장면을 추가했어요`라는 상태 변화를
보조기기에 알린다. 모바일 목차 시트에서는 추가 또는 선택 뒤 시트를 닫고 편집기로 이동한다.
데스크톱의 인라인 목차는 열린 상태를 유지한다.

## 데이터 흐름

1. 사용자가 `새 장면 추가`를 활성화한다.
2. 집필 워크스페이스가 충돌하지 않는 장면 ID를 생성한다.
3. Manuscript 도메인이 다음 장 번호를 계산하고 새 활성 장면을 포함한 원고를 반환한다.
4. 자동 저장 초안이 즉시 갱신되어 목차, 헤더, 편집기가 함께 다시 렌더링된다.
5. 기존 자동 저장 지연 뒤 전체 원고와 기대 리비전을 `saveManuscript`에 전달한다.
6. 저장 성공 시 서버 원고와 새 리비전을 채택한다. 새로고침 뒤에도 저장된 장면과 활성 장면이
   복원된다.

기존 장면을 선택할 때도 같은 초안 갱신 경로를 사용한다. 활성 장면 변경은 Manuscript 상태의
일부이므로 기존 전체 원고 저장 흐름으로 영속화한다.

## 저장 상태와 오류 처리

- 추가 뒤 저장 대기, 저장 중, 저장 완료 표시는 기존 자동 저장 표시기를 사용한다.
- 저장 실패 시 새 장면과 이후 입력 내용은 로컬 초안에 유지한다.
- 사용자는 기존 `원고 저장 다시 시도` 동작으로 동일한 최신 초안을 다시 저장한다.
- 리비전 충돌 상태에서는 추가 버튼을 비활성화하여 해결 전 구조 변경이 더 누적되지 않게 한다.
- 장면 선택은 로컬 초안을 잃지 않으므로 저장 대기 중에도 허용한다.

## 구조 충돌 해결

기존 충돌 흐름은 서버에도 존재하는 한 장면의 본문 비교를 전제로 한다. 새 활성 장면은 서버에
아직 없을 수 있으므로 구조 충돌을 별도로 판별하고 해결해야 한다.

자동 저장 계층은 마지막으로 승인된 원고, 현재 로컬 초안, 다시 조회한 서버 최신 원고를
구분한다. 로컬에서 마지막 승인본 이후 추가되었고 서버에는 없는 장면을 로컬 전용 장면으로
분류한다. 구조 충돌 화면은 현재 초안을 보존한 채 다음 선택을 제공한다.

- **내 새 장면 유지:** 서버 최신 원고의 기존 장면과 순서를 기준으로 삼고 로컬 전용 장면을
  뒤에 합친다. 합친 원고의 활성 장면은 현재 로컬 활성 장면으로 유지하고 서버 최신 리비전으로
  다시 저장한다.
- **서버 최신본 적용:** 로컬 전용 장면과 미저장 내용을 버리고 서버 최신 원고와 리비전을
  채택한다.

동일한 장면 ID가 서버에도 별개로 생겼거나, 병합 대상의 장 번호가 서버 최신 변경과 충돌하면
자동으로 덮어쓰지 않는다. 해결 실패 상태와 재시도 수단을 표시하고 로컬 초안을 유지한다.
로컬과 서버가 같은 기존 장면 본문을 모두 변경한 경우에는 기존 장면 비교 흐름을 사용한다.
구조 충돌 해결은 기존 작업공간 조회와 전체 원고 저장 작업만 사용한다.

## 접근성 및 반응형 동작

- 추가 버튼은 `새 장면 추가`라는 접근 가능한 이름을 유지한다.
- 각 장면 버튼은 장 번호와 제목을 함께 읽을 수 있는 이름을 제공한다.
- 활성 장면은 색상만이 아니라 `aria-current`와 기존 선택 스타일로 전달한다.
- 추가 성공 알림은 비방해성 live region을 사용한다.
- 저장 실패와 충돌 실패는 기존 `role="alert"` 패턴과 명시적 재시도 버튼을 사용한다.
- 모바일에서 목차가 닫힌 뒤 편집기 포커스로 이동하고, 데스크톱에서는 목차 레이아웃을
  유지한다.

## 테스트 전략

### 도메인 단위 테스트

- 빈 장면의 기본값과 다음 장 번호 계산
- 장면 배열 끝 추가와 새 활성 장면 설정
- 입력 원고 및 기존 장면 불변성
- 중복 장면 ID 거부
- 존재하는 장면 선택과 없는 장면 선택 거부

### UI 및 통합 테스트

- 추가 버튼 한 번당 장면 하나 생성
- 새 목차 항목과 활성 스타일, 편집기 및 헤더 갱신
- 기존 장면 재선택과 선택 범위 초기화
- 자동 저장 요청에 기존 장면과 새 장면이 모두 포함됨
- 저장 실패 뒤 새 장면 및 입력 보존과 재시도
- 충돌 중 추가 비활성화
- 구조 충돌에서 내 새 장면 유지와 서버 최신본 적용
- ID 또는 장 번호 충돌 시 로컬 초안 보존
- 키보드 조작, live region, 포커스 이동, 모바일 목차 닫힘

### E2E

프런트엔드 구현 절차에 따라 별도 E2E 계획을 만든 뒤, 장면 추가부터 입력·자동 저장·재접속
복원까지의 핵심 흐름과 저장 실패 복구를 Playwright로 검증한다.

## 완료 기준

- 기존 장면과 본문을 보존하면서 새 장면이 정확히 하나 추가된다.
- 새 장면은 마지막 목차 항목이자 활성 장면이며 빈 편집기가 즉시 입력 가능하다.
- 사용자는 모든 기존 장면과 새 장면 사이를 선택해 이동할 수 있다.
- 활성 장면의 번호와 제목이 목차, 헤더, 편집기에서 일치한다.
- 추가한 장면과 입력 내용이 자동 저장, 재시도, 구조 충돌 해결 과정에서 보존된다.
- 저장 뒤 새로고침해도 장면 목록과 활성 장면이 복원된다.
- 도메인 계약과 구현 및 테스트가 동일한 장면 추가 규칙을 설명한다.
- 장면 이름 변경, 순서 변경, 삭제 또는 새 API 엔드포인트가 도입되지 않는다.
