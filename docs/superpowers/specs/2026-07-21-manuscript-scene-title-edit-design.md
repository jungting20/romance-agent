# 원고 장면 제목 수정 설계

## 배경

Muse의 Manuscript는 장면 식별자, 제목, 장 번호, 본문과 Story Bible 문맥 참조를 가진 장면 배열을
소유한다. 현재 집필 작업공간은 활성 장면의 제목을 편집기, 작업공간 헤더와 원고 목차에 표시하지만
사용자가 제목을 바꿀 수는 없다.

이 기능은 별도 장·부 계층을 만들지 않고 기존 `Scene.title`만 수정한다. 티켓 `#3 원고 장면 추가와
전환`이 제공하는 장면 선택과 활성 장면 동기화를 선행 기반으로 사용하며 그 기능을 중복 구현하지 않는다.

## 목표

사용자는 집필 편집기에서 현재 장면 제목을 인라인으로 수정하고, 편집기·작업공간 헤더·원고 목차에서
같은 제목을 즉시 확인할 수 있어야 한다. 제목 변경은 기존 전체 원고 자동 저장, 실패 재시도와 리비전
충돌 흐름을 통해 원고 본문과 함께 보존되어야 한다.

## 선행 조건

구현을 시작하기 전에 다음 티켓의 장면 추가와 선택 동작이 기준 브랜치에 통합되어 있어야 한다.

- ticket-worker `#3 원고 장면 추가와 전환`
- `docs/superpowers/specs/2026-07-16-manuscript-scene-add-design.md`
- `docs/superpowers/plans/2026-07-16-manuscript-scene-add.md`

선행 조건이 충족되지 않았으면 이 기능에서 장면 선택, 활성 표시, 구조 충돌 기반을 복사하지 않고 구현을
중단해 선행 티켓 미충족을 보고한다.

## 범위

### 포함

- 활성 장면 제목 옆의 `장면 제목 수정` 진입점
- 기존 제목을 채운 인라인 단일 입력란
- Enter와 포커스 이탈을 통한 변경 확정
- Escape를 통한 변경 취소
- 제목 앞뒤 공백 제거와 공백 제목 거부
- 편집기, 작업공간 헤더와 원고 목차의 제목 동기화
- 기존 전체 원고 자동 저장, 실패 재시도와 리비전 충돌 흐름 사용
- Manuscript 도메인 계약, 단위 테스트, 컴포넌트·화면 통합 테스트와 E2E 검증

### 제외

- 장, 부, 화 또는 막을 별도 컨테이너로 만드는 계층형 목차
- 장면 번호 수정
- 장면 제목 일괄 변경
- 장면 순서 변경, 이동, 복제 또는 삭제
- 장면 계획과 초고 상태
- 프로젝트 제목 수정
- AI 제목 생성 또는 추천
- 새 장면 전용 또는 제목 전용 API 작업

## 설계 접근

Manuscript 도메인에 제목 수정 순수 연산을 추가하고, 집필 편집기의 인라인 편집 컴포넌트가 확정된
제목만 이 연산으로 전달한다. 변경된 전체 Manuscript는 기존 자동 저장 초안으로 채택한다.

UI 내부에서만 제목을 바꾸는 방식은 도메인 검증과 목차·헤더 동기화를 우회하므로 사용하지 않는다.
제목 전용 API도 기존 전체 원고 저장 경계와 중복되므로 추가하지 않는다.

## 도메인 설계

Manuscript는 다음 연산을 제공한다.

```ts
updateSceneTitle(manuscript: Manuscript, sceneId: string, title: string): Manuscript
```

연산 규칙은 다음과 같다.

- `sceneId`는 입력 원고에 존재하는 장면을 가리켜야 한다.
- 입력 제목은 앞뒤 공백을 제거한 뒤 비어 있지 않아야 한다.
- 유효한 제목은 정규화한 값으로 해당 장면의 `title`만 변경한다.
- 장면 식별자, 장 번호, 본문, 관련 인물·세계관 참조와 다른 장면은 보존한다.
- 입력 Manuscript와 기존 Scene 객체를 직접 변경하지 않고 새 Manuscript를 반환한다.
- 정규화한 제목이 현재 제목과 같으면 입력 Manuscript를 그대로 반환할 수 있다.
- 존재하지 않는 장면이나 공백 제목은 전체 요청을 거부하고 부분 결과를 반환하지 않는다.

공백이 아닌 제목에 새 최대 길이 제한을 도입하지 않는다. 이 규칙은 제목 수정 유스케이스의 입력과
결과를 검증하며, 기존 저장 데이터를 일괄 재작성하거나 별도 마이그레이션하지 않는다.

`docs/domains/manuscript.md`에는 장면 제목 수정 유스케이스, 제목 정규화와 비어 있지 않은 수정 결과,
불변 갱신 규칙을 구현과 같은 변경에서 추가한다.

## UI와 상호작용

### 기본 상태

`ManuscriptEditor`는 장면 제목 옆에 아이콘 버튼을 표시한다. 버튼의 접근 가능한 이름은 정확히
`장면 제목 수정`이다. 제목 자체는 기존 heading 의미를 유지한다.

### 편집 상태

버튼을 활성화하면 제목 heading 위치를 단일 텍스트 입력란으로 바꾸고 현재 제목 전체를 선택한 채 포커스를
이동한다. 입력란의 접근 가능한 이름은 `장면 제목`이다.

- Enter: 제목 변경을 확정한다.
- 포커스 이탈: 제목 변경을 확정한다.
- Escape: 입력을 폐기하고 기존 제목으로 돌아간다.
- 확정 성공: heading과 수정 버튼을 복원하고 `장면 제목을 저장할 준비가 되었어요.`를 비방해성 상태로
  알린다.
- 공백 제목: 도메인 변경을 호출하지 않고 입력란을 유지하며 `장면 제목을 입력해 주세요.`를 필드 오류로
  표시한다.

별도 확인 대화상자는 표시하지 않는다. 사용자가 수정 버튼을 선택하고 Enter 또는 포커스 이탈로 확정한
행위가 충분한 의사 표현이다.

### 상태 동기화

확정된 제목은 하나의 Manuscript 초안에서 파생하므로 다음 위치가 같은 렌더링 주기에서 갱신된다.

- 편집기 장면 heading
- 작업공간 헤더의 `N장 · 제목`
- 원고 목차의 활성 장면 항목

장면을 전환하면 편집 상태를 종료하고 새 활성 장면 제목을 표시한다. Escape로 취소한 값은 Manuscript
초안이나 자동 저장 요청에 포함하지 않는다.

## 데이터 흐름과 저장

1. 사용자가 `장면 제목 수정`을 선택한다.
2. 인라인 편집기는 현재 활성 장면 제목을 로컬 입력 상태로 제공한다.
3. 사용자가 Enter 또는 포커스 이탈로 확정한다.
4. 작업공간 기능이 `updateSceneTitle`을 호출한다.
5. 성공한 새 Manuscript를 기존 자동 저장 초안으로 채택한다.
6. 편집기, 헤더와 목차가 같은 제목을 다시 렌더링한다.
7. 기존 800ms 자동 저장 지연 뒤 전체 Manuscript와 기대 리비전을 `saveManuscript`에 전달한다.
8. 저장 성공 시 서버 Manuscript와 새 리비전을 승인된 기준 상태로 채택한다.

`Scene.title`과 전체 Manuscript 저장은 현재 OpenAPI 계약에 이미 존재한다. 새 경로, 메서드,
operationId, 요청·응답 필드 또는 오류 코드를 추가하지 않는다. 구현 시작 시 Main은 현재
`saveManuscript` 기준을 확인해 프런트엔드와 리뷰에 같은 기준을 제공한다.

## 저장 실패와 충돌

- 일반 저장 실패 시 수정한 제목과 원고 본문을 최신 로컬 Manuscript 초안에 유지한다.
- 사용자는 기존 `원고 저장 다시 시도`로 같은 최신 초안을 다시 저장한다.
- 리비전 충돌이 발생하면 기존 충돌 화면과 로컬·서버 선택 규칙을 사용한다.
- 충돌 상태에서는 새 제목 편집을 시작하거나 확정할 수 없다.
- 제목을 입력하는 동안 이전 자동 저장 요청이 충돌하면 입력값을 화면에 보존하고 확정을 비활성화한다.
  충돌 해결 후 현재 활성 장면이 유지되면 사용자가 같은 입력값을 검토하고 확정할 수 있다.
- 서버 최신본 적용으로 활성 장면이 바뀌거나 대상 장면이 사라지면 보존한 입력값을 자동 적용하지 않고
  현재 서버 장면 제목을 표시한다.

제목 전용 자동 병합 규칙은 만들지 않는다. 충돌 해결이 완료되기 전에 제목을 서버 원고에 덮어쓰지 않는다.

## 접근성과 반응형 동작

- heading과 수정 버튼은 각각 제목 정보와 동작 의미를 유지한다.
- 편집 시작 시 입력란으로 포커스를 이동하고 현재 제목을 선택한다.
- Enter, Escape와 포커스 이탈을 키보드와 포인터 모두에서 사용할 수 있다.
- 필드 오류는 입력란과 연결하고 오류 메시지를 보조기기가 인식할 수 있게 한다.
- 제목 확정, 취소와 저장 상태를 색상에만 의존하지 않는다.
- 모바일과 데스크톱에서 같은 인라인 상호작용을 사용하며 작은 화면에서 버튼이 제목을 가리지 않게 한다.
- 자동 저장 상태는 기존 작업공간 상단 표시기를 계속 사용한다.

## 테스트 전략

### 도메인 단위 테스트

- 유효한 제목의 앞뒤 공백 제거
- 해당 장면 제목만 변경하고 다른 필드와 장면 보존
- 입력 Manuscript와 기존 Scene 불변성
- 현재 제목과 같은 정규화 결과의 멱등성
- 공백 제목과 존재하지 않는 장면 거부

### 컴포넌트 테스트

- 수정 버튼으로 입력란을 열고 기존 제목 전체 선택
- Enter와 포커스 이탈을 통한 확정
- Escape를 통한 취소
- 공백 제목 오류와 입력 상태 유지
- 충돌 상태에서 시작·확정 비활성화
- 장면 전환과 서버 최신본 적용 뒤 입력 상태 정리
- 접근 가능한 이름, 오류 연결과 포커스 복원

### 화면 및 저장 통합 테스트

- 확정 직후 편집기, 헤더와 목차 제목 동기화
- 전체 원고 저장 요청에 수정 제목과 기존 본문·참조가 함께 포함됨
- 저장 성공과 새로고침 뒤 제목 복원
- 저장 실패 뒤 로컬 제목 유지와 재시도
- 리비전 충돌에서 로컬 초안 유지 또는 서버 최신본 적용
- 모바일과 데스크톱의 동일한 상호작용

### E2E

프런트엔드 구현과 읽기 전용 애플리케이션 리뷰가 끝난 뒤 프로젝트 E2E 절차를 따른다. 최소 흐름은
제목 편집, Enter 확정, 세 표시 위치 동기화, 자동 저장과 재접속 복원, Escape 취소와 공백 오류를
검증한다. 저장 실패와 충돌 복구도 사용자에게 보이는 결과로 확인한다.

## 완료 기준

- 사용자가 현재 장면 제목을 인라인으로 수정하거나 취소할 수 있다.
- 공백 제목은 원고에 반영되거나 저장되지 않는다.
- 확정된 제목은 편집기, 작업공간 헤더와 원고 목차에서 일치한다.
- 제목 변경이 장면 식별자, 번호, 본문과 Story Bible 문맥 참조를 변경하지 않는다.
- 저장 실패와 충돌 중에도 원고와 확정된 로컬 제목이 조용히 사라지지 않는다.
- 저장 뒤 새로고침해도 수정한 제목이 복원된다.
- 구현, 테스트와 `docs/domains/manuscript.md`가 같은 제목 수정 규칙을 설명한다.
- 장·부 계층, 장면 번호·순서·삭제·계획·상태, 프로젝트 제목, AI와 새 API가 도입되지 않는다.

## 승인 기록

사용자는 2026-07-21에 별도 장·부 계층을 도입하지 않고 기존 `Scene.title`만 수정하는 범위를 선택했다.
세 가지 UI 진입 방식 중 편집기 제목 옆 수정 버튼과 인라인 편집을 선택했으며, 기존 전체 Manuscript
자동 저장과 티켓 #3의 장면 전환 기반을 재사용하는 설계를 승인했다.
