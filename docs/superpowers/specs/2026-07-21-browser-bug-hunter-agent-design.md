# 브라우저 버그 탐색 Sub-agent 설계

## 목적

프로젝트에 지정된 화면과 사용자 흐름을 실제 브라우저로 탐색하고, 객관적으로 재현되는 UI·UX 결함을 직접 `ticket-worker`에 등록하는 프로젝트 전용 `bug-hunter` sub-agent를 추가한다. 사용자는 탐색 결과 전체를 한글 HTML 보고서로 열람할 수 있어야 한다.

기존 `frontend-review`는 완료된 구현의 규칙·계약 적합성을 검토하고, `playwright_test_planner`는 승인된 구현의 E2E 테스트 계획을 만든다. `bug-hunter`는 구현 여부와 무관하게 실행 중인 제품을 사용자처럼 조작해 결함을 발견하고 재현하는 역할만 맡는다.

## 선택한 접근

`.codex/agents/bug-hunter.toml`에 하나의 프로젝트 전용 custom agent를 정의한다. 이 agent가 브라우저 탐색, 재현 확인, 중복 검사, 티켓 문서 작성, 제한된 커밋, 티켓 등록, 한글 HTML 보고서 생성을 순서대로 수행한다.

탐색과 등록을 서로 다른 agent로 분리하는 방식은 권한 격리가 더 강하지만 작은 탐색 실행마다 조정 비용이 크다. 기존 Playwright test planner를 확장하는 방식은 테스트 계획과 제품 결함 판정 책임을 섞으므로 사용하지 않는다.

## 호출 계약

메인 agent는 `bug-hunter`를 호출할 때 다음 정보를 반드시 제공한다.

- 대상 라우트 또는 화면 진입점
- 깊게 탐색할 사용자 흐름
- 시작 데이터와 인증 상태
- 데스크톱·모바일 등 필수 뷰포트
- 관련 요구사항, 도메인 계약, 승인된 UI 계획 또는 수용 기준
- 허용된 서버 시작·검증 명령
- 탐색에서 제외할 동작과 데이터 변경 제한

대상 화면이나 사용자 흐름이 빠지면 `bug-hunter`는 탐색을 시작하지 않고 메인 agent에 누락을 보고한다. 할당된 흐름을 이해하는 데 필요한 인접 화면은 볼 수 있지만, 별도의 기능이나 애플리케이션 전체 감사로 범위를 확장하지 않는다.

## Agent 구성

agent 이름은 `bug-hunter`로 하고 설명은 “지정된 화면과 사용자 흐름을 브라우저에서 깊게 탐색하여 재현 가능한 UI·UX 결함을 찾아 직접 티켓으로 등록할 때 사용”으로 정의한다.

제품 코드를 수정하지 못하도록 기본 작업은 읽기 중심으로 제한하되, 다음 경로에만 산출물을 작성할 수 있도록 developer instructions에서 소유권을 제한한다.

- `docs/bug-reports/**`
- 발견한 버그를 위한 `docs/superpowers/specs/**`
- 발견한 버그를 위한 `docs/superpowers/plans/**`
- 프로젝트 로컬 `ticket-worker` 데이터

agent 자신, `.codex/agents/**`, `frontend/**`, `backend/**`, `docs/api/openapi.yaml`, `docs/domains/**`는 수정하지 않는다. Git 커밋은 자신이 새로 만든 버그 설계·계획 문서와 HTML 보고서·증거 이미지만 명시적으로 스테이징한다. 기존 변경은 스테이징, 커밋, 복원 또는 삭제하지 않는다.

브라우저 탐색에는 기존 Playwright Test MCP 서버 구성을 재사용한다. 브라우저 탐색과 증거 수집에 필요한 도구만 활성화하고 테스트 생성·수정 도구는 제공하지 않는다. 소스 위치를 조사할 때 `.codegraph/`가 있으면 저장소 지침에 따라 CodeGraph를 먼저 사용한다.

## 탐색 절차

1. 루트 `AGENTS.md`, 해당 애플리케이션의 `AGENTS.md`와 코딩 규칙, 관련 도메인 계약, 할당된 수용 기준을 읽는다.
2. 현재 Git 상태와 기존 `ticket-worker` 목록을 읽어 사용자 변경과 기존 티켓 기준선을 기록한다.
3. 지정된 시작 상태로 브라우저를 준비하고 할당된 사용자 흐름의 정상 경로를 먼저 확인한다.
4. 같은 흐름 안에서 경계값, 빈 상태, 오류·복구, 키보드 조작, 포커스, 반응형 레이아웃을 탐색한다.
5. 의심 현상을 깨끗한 시작 상태에서 두 번 재현한다. 두 번 모두 같은 관찰 결과가 나와야 확정 버그가 된다.
6. 확정 버그에 대해 최소 재현 절차, 기대 결과, 실제 결과, 사용자 영향, 환경, 관련 브라우저 증거를 기록한다.
7. 필요한 경우에만 관련 소스를 읽어 가장 좁은 관련 파일·심볼과 추정 원인을 기록한다. 확인되지 않은 원인을 사실로 단정하지 않는다.
8. 기존 티켓의 라우트, 관찰 가능한 현상, 영향, 계획 경로를 비교해 중복을 제거한다.
9. 등록 대상마다 하나의 설계 문서와 구현 계획을 작성하고 자체 검토한 뒤 해당 문서만 커밋한다.
10. `ticket-worker`에 티켓을 등록하고 반환된 ID, `ready` 상태, 문서 경로, 구현 프롬프트를 검증한다.
11. 탐색 전체 결과를 한글 HTML 보고서로 작성하고 증거 이미지와 함께 별도 커밋한다.
12. 메인 agent에 보고서 경로, 등록 티켓, 제외 항목, 검사 결과와 제약을 반환한다.

## 버그 판정과 등록 게이트

다음 조건을 모두 충족해야 티켓으로 등록한다.

- 할당된 화면과 사용자 흐름 안에서 발생한다.
- 깨끗한 시작 상태에서 두 번 재현된다.
- 기대 결과가 도메인 계약, 수용 기준, 접근성 요구, 명시적 제품 동작 또는 안정된 기존 동작으로 뒷받침된다.
- 실제 결과와 사용자 영향이 구체적이다.
- 기존 티켓과 관찰 가능한 실패가 중복되지 않는다.
- 수정 방향이 제품 코드 변경으로 해결 가능한 하나의 결함으로 좁혀진다.

주관적 디자인 선호, 단순 개선 아이디어, 사용자가 볼 수 없는 코드 스타일, 재현되지 않는 현상, 요구사항 자체의 미해결 제품 결정은 등록하지 않는다. 등록하지 않은 관찰도 HTML 보고서의 “제외한 관찰”에 이유와 함께 남긴다.

각 버그는 `Blocking`, `High`, `Medium`, `Low` 중 하나로 분류하되, 심각도와 관계없이 위 게이트를 모두 통과한 객관적 결함은 직접 등록한다. 여러 증상이 하나의 원인과 수정 단위를 공유하면 한 티켓으로 묶고, 독립적으로 수정·검증할 수 있으면 분리한다.

## 티켓 문서와 등록

버그별 설계 문서는 `docs/superpowers/specs/YYYY-MM-DD-<bug-slug>-design.md`, 구현 계획은 `docs/superpowers/plans/YYYY-MM-DD-<bug-slug>.md`에 작성한다. 문서는 문제, 재현 절차, 기대·실제 결과, 사용자 영향, 범위, 제외 범위, 수용 기준, 검증 방법을 포함한다. 구현 계획은 실패 재현 테스트, 최소 수정, 집중 검증, 전체 애플리케이션 검증, 필수 리뷰 순서를 포함한다.

티켓 제목과 요약은 한글로 작성한다. 구현 프롬프트는 두 문서 경로를 모두 명시하고 `feature-development`의 전체 구현·검토·검증 절차를 따르며, brainstorming과 writing-plans를 다시 수행하지 말라고 지시한다.

`ticket-worker`가 미초기화 상태일 때만 한 번 초기화한다. 등록 후 JSON 응답에서 `ready` 상태, 제목, 설계·계획 경로, 프롬프트를 검증한다. 중복이 발견되면 새 티켓을 만들지 않고 기존 티켓 ID를 보고서에 연결한다. 문서 커밋이나 등록이 실패하면 후속 단계를 성공으로 표시하지 않고 실제 상태를 보고한다.

## 한글 HTML 보고서

탐색 실행마다 다음 경로에 보고서 하나를 만든다.

```text
docs/bug-reports/YYYY-MM-DD-HHmm-<scope-slug>.html
```

필요한 스크린샷은 같은 이름의 자산 디렉터리에 저장한다.

```text
docs/bug-reports/assets/YYYY-MM-DD-HHmm-<scope-slug>/
```

HTML은 빌드나 외부 CDN 없이 로컬 브라우저에서 바로 열 수 있어야 한다. `lang="ko"`, UTF-8 메타데이터, 반응형 인라인 CSS, 의미 있는 제목 구조, 키보드로 접근 가능한 링크, 색상 외 심각도 텍스트, 인쇄 스타일을 포함한다. JavaScript는 사용하지 않는다.

보고서의 사용자 대상 문장은 모두 한글로 작성한다. 파일 경로, 코드 심볼, 명령어, URL, 티켓 상태값처럼 정확한 원문이 필요한 기술 식별자만 그대로 표기한다.

보고서는 다음 순서로 구성한다.

1. 제목과 생성 시각
2. 탐색 범위와 제외 범위
3. 실행 환경, 시작 상태, 뷰포트
4. 발견·등록·중복·제외 건수 요약
5. 등록한 버그 목록
6. 버그별 심각도, 재현 절차, 기대 결과, 실제 결과, 사용자 영향, 증거, 관련 파일·심볼, 티켓 ID
7. 기존 티켓과 중복된 관찰
8. 재현 실패·주관적 개선 등 제외한 관찰과 제외 이유
9. 수행한 명령과 결과
10. 탐색 제약과 확인하지 못한 항목

각 버그에는 `bug-001` 형식의 안정적인 HTML 앵커를 부여한다. HTML에서 설계·계획 문서로 상대 링크를 제공하고, 각 티켓 문서에는 해당 보고서 파일과 버그 앵커를 기록해 양방향 추적성을 유지한다. 스크린샷에는 한글 캡션과 대체 텍스트를 제공하며, 민감 정보가 포함된 이미지는 저장하지 않는다.

버그가 하나도 없어도 보고서를 생성한다. 이 경우 탐색한 흐름, 확인한 상태, 등록 0건, 제외 관찰, 제약을 명시하며 “버그가 없다”고 일반화하지 않는다.

## 저장소 지침 변경

루트 `AGENTS.md`의 sub-agent 절차에 `bug-hunter` 호출 조건과 할당 필수 입력을 추가한다. 메인 agent는 대상 라우트와 흐름을 경계 짓고, 동시에 같은 화면을 수정하는 구현 agent가 없을 때 탐색을 시작하며, 반환된 직접 등록 티켓과 보고서를 최종 검토한다.

`bug-hunter`가 직접 등록 권한을 갖더라도 메인 agent의 통합 책임은 유지된다. 메인 agent는 중복이나 잘못 등록된 티켓을 발견하면 구체적 사유를 기록하고 취소할 수 있으며, 제품 구현을 시작하기 전 티켓의 범위와 문서를 확인한다.

## 검증

- TOML을 파싱해 필수 `name`, `description`, `developer_instructions`와 MCP 설정을 확인한다.
- agent 지침에 읽기 범위, 쓰기 허용 경로, 금지 경로, 재현 2회, 중복 검사, 티켓 검증, 한글 HTML 요구가 모두 포함됐는지 정적 검증한다.
- 샘플 할당을 사용한 dry run에서 제품 코드와 티켓 DB를 변경하지 않고 탐색 계획, 등록 게이트, 예상 보고서 경로를 반환하게 한다.
- 샘플 HTML fixture 또는 생성 예시를 검사해 `lang="ko"`, UTF-8, 필수 섹션, 상대 링크, 이미지 대체 텍스트, 외부 CDN·JavaScript 부재를 확인한다.
- 기존 백엔드·프론트엔드·OpenAPI·리뷰 agent 정의가 변경되지 않았는지 확인한다.

## 수용 기준

1. 메인 agent가 지정된 화면과 사용자 흐름을 `bug-hunter`에 위임할 수 있다.
2. agent는 할당 범위에서 실제 브라우저를 사용하고 확정 버그를 두 번 재현한다.
3. 중복이 아닌 객관적 버그마다 구현 가능한 설계·계획 문서를 만들고 `ready` 티켓을 직접 등록한다.
4. agent는 제품 코드, 도메인 계약, OpenAPI 또는 다른 agent 정의를 수정하지 않는다.
5. 매 실행마다 탐색 결과 전체를 한글 HTML 보고서로 생성하며 버그가 0건이어도 보고서를 남긴다.
6. 보고서, 티켓, 설계·계획 문서 사이에 추적 가능한 링크가 있다.
7. 메인 agent는 등록 결과와 보고서를 검토하고 이후 구현 여부를 결정할 수 있다.
