# 원고 장면 추가 E2E 테스트 계획

## 목적과 범위

대상은 `/projects/silver-garden/write`의 승인된 장면 추가·선택·자동 저장·구조 충돌 흐름이다. 이 계획은 새 장면 이름 변경, 순서 변경, 삭제, 중첩 목차 또는 새 API를 다루지 않는다. 모든 저장은 기존 `PUT /api/manuscripts/silver-garden-manuscript`, 구조 충돌의 서버 최신본 조회는 기존 `GET /api/projects/silver-garden/workspace`만 사용한다.

## 공통 시작 상태와 네트워크 제어

- 각 테스트는 `serviceWorkers: "block"`인 새 BrowserContext/Page에서 시작해 앱의 개발용 MSW가 `page.route` 응답을 가로채지 않게 한다. `page.addInitScript`로 `crypto.randomUUID()`가 테스트별 고정 ID(예: `e2e-local-scene`)를 반환하게 한다.
- 각 테스트는 탐색 전에 `page.route`를 등록한다. Service Worker의 공유 메모리 상태에 의존하지 말고 아래 fixture를 테스트 안에서 깊은 복사하여 사용한다.
- 초기 workspace:
  - 프로젝트 `silver-garden`, 원고 `silver-garden-manuscript`, `manuscriptRevision: 1`.
  - 장면은 `silver-garden-scene-1` 하나이며 제목 `비가 그친 뒤의 정원`, 장 번호 1, 승인된 초기 본문, 활성 장면도 이 장면이다.
  - 성공한 PUT은 요청의 전체 `manuscript`를 서버 fixture에 저장하고 revision을 1 증가시킨 뒤 같은 원고와 새 revision을 응답한다. 이후 GET은 갱신된 fixture를 반환한다.
- 요청 기록에는 메서드, URL, `expectedRevision`, 전체 `manuscript`를 남긴다. 대기는 고정 sleep 대신 요청 promise와 사용자 표시(`role="status"`, `role="alert"`, dialog)를 사용한다. 제품의 800ms autosave debounce는 Playwright의 실제 시간으로 기다리되 PUT 관찰을 종료 조건으로 삼는다.
- 데스크톱 기본 viewport는 `1280 × 900`, 모바일은 `375 × 812`로 한다.
- 공통 초기 확인은 `textbox`/`원고 본문`, `button`/`1장 비가 그친 뒤의 정원`, `status`/`자동 저장됨`이다. 장면 버튼은 accessible name과 `aria-current`로 판정한다.
- 테스트 간 독립성을 위해 서버 fixture, 요청 배열, UUID, route handler를 `beforeEach`에서 재생성하고, 테스트 종료 후 page/context와 route를 폐기한다. 한 테스트가 만든 revision이나 장면을 다른 테스트가 재사용하지 않는다.

## 시나리오 1 — 새 장면을 즉시 추가하고 편집기 포커스와 live 알림을 제공한다

**추적:** 완료 기준 1, 2, 4; 접근성 요구(추가 버튼 이름, 활성 상태, live region, 포커스).

1. 데스크톱 초기 workspace를 연다. 장면 버튼 `1장 비가 그친 뒤의 정원`이 `aria-current="true"`인지 확인한다.
2. `button` 이름 `새 장면 추가`를 한 번 클릭한다.
3. 목차에 `button` 이름 `2장 제목 없는 장면`이 정확히 하나 생기고 마지막 장면이며 `aria-current="true"`인지 확인한다. 첫 장면은 남아 있고 더 이상 current가 아니어야 한다.
4. `heading` 이름 `제목 없는 장면`, 헤더 텍스트 `2장 · 제목 없는 장면`, 빈 `textbox` 이름 `원고 본문`이 같은 활성 장면을 나타내는지 확인한다.
5. `원고 본문`이 focused element인지 확인한다.
6. `[aria-live="polite"]` 영역이 `2장 장면을 추가했어요`를 포함하는지 확인한다. 시각 노출 여부가 아니라 live region의 DOM/접근성 속성과 텍스트를 검증한다.

**성공 기준:** 한 번의 동작으로 기본값을 가진 장면 하나만 추가되고, 편집기가 즉시 입력 가능하며 장면 번호·제목·활성 상태와 알림이 일치한다. 초기 장면이나 본문이 사라지거나 추가 PUT 외의 새 API가 호출되면 실패다.

## 시나리오 2 — 새 장면에 입력한 전체 원고를 자동 저장하고 새로고침 후 복원한다

**추적:** 완료 기준 2, 5, 6; 데이터 흐름 4–6.

1. 데스크톱 초기 workspace에서 새 장면을 추가하고 `원고 본문`에 `온실 문이 천천히 열렸다.`를 입력한다.
2. 표시기가 `편집 중` 또는 `저장 중`을 거쳐 accessible name `자동 저장됨`으로 돌아올 때까지 PUT을 기다린다.
3. 첫 PUT의 `expectedRevision`이 1이고 `manuscript.scenes`가 두 장면 전체를 포함하는지 확인한다. 첫 장면의 모든 필드는 초기값 그대로이고, 두 번째 장면은 ID `silver-garden-scene-e2e-local-scene`, 제목 `제목 없는 장면`, 장 번호 2, 입력 본문, 빈 관련 인물/세계관 배열이어야 한다. `activeSceneId`는 두 번째 장면 ID여야 한다.
4. PUT 성공 응답으로 서버 fixture가 revision 2와 요청 원고를 채택하게 한 뒤 페이지를 reload한다.
5. reload GET 이후 `2장 제목 없는 장면`이 current이고, 헤더가 `2장 · 제목 없는 장면`, 편집기 값이 `온실 문이 천천히 열렸다.`, 상태가 `자동 저장됨`인지 확인한다.

**성공 기준:** 기존 전체-Manuscript 계약으로 입력과 활성 장면이 저장되고 실제 reload 뒤 복원된다. 장면 일부만 보내거나 첫 장면을 잃거나 revision 1을 재사용하면 실패다.

## 시나리오 3 — 기존 장면을 다시 선택해도 새 장면과 입력을 잃지 않는다

**추적:** 완료 기준 1, 3, 4, 5.

1. 데스크톱 초기 workspace에서 새 장면을 추가하고 `원고 본문`에 `두 번째 장면의 로컬 문장`을 입력한다. 자동 저장 성공을 기다린다.
2. `button` 이름 `1장 비가 그친 뒤의 정원`을 클릭한다.
3. 첫 장면 버튼이 current가 되고 헤더가 `1장 · 비가 그친 뒤의 정원`, 편집기 값이 초기 첫 장면 본문인지 확인한다. `2장 제목 없는 장면` 버튼은 계속 존재해야 한다.
4. `button` 이름 `2장 제목 없는 장면`을 다시 클릭한다.
5. 두 번째 장면이 current이고 편집기 값이 `두 번째 장면의 로컬 문장`으로 돌아오는지 확인한다. 저장 요청에는 두 장면이 모두 남아 있어야 한다.
6. reload 후에도 두 번째 장면을 선택하면 같은 입력이 복원되는지 확인한다.

**성공 기준:** 선택은 `activeSceneId`만 전환하며 장면 내용·순서·개수를 바꾸지 않는다. 선택 때문에 새 장면이나 그 본문이 사라지면 실패다.

## 시나리오 4 — 모바일 원고 Sheet에서 추가 후 Sheet를 닫고 편집기로 이동한다

**추적:** 완료 기준 2, 3; 모바일 접근성/반응형 요구.

1. viewport `375 × 812`의 초기 workspace를 연다.
2. `tab` 이름 `원고 보기`를 클릭하고 `dialog` 이름 `원고 보기`가 열렸는지 확인한다.
3. dialog 안의 `button` 이름 `새 장면 추가`를 한 번 클릭한다.
4. `dialog` 이름 `원고 보기`가 닫히고 `textbox` 이름 `원고 본문`이 focused element가 될 때까지 기다린다.
5. 다시 `원고 보기`를 열어 `2장 제목 없는 장면`이 current이고 첫 장면도 남아 있는지 확인한 뒤 Sheet를 닫는다.

**성공 기준:** 모바일에서 추가 또는 그 결과 확인 뒤 편집기로 자연스럽게 돌아오며, 데스크톱 전용 인라인 목차를 가정하지 않는다. Sheet가 남거나 focus가 trigger/body에 머물면 실패다.

## 시나리오 5 — 일반 저장 실패와 재시도 중 로컬 새 장면을 보존한다

**추적:** 완료 기준 5, 6; 저장 상태와 오류 처리.

**네트워크:** 첫 PUT은 `500`과 `{ code: "INTERNAL_ERROR", ... }`, 두 번째 PUT은 성공 revision 2를 반환한다. GET은 성공한 두 번째 PUT 이후 그 원고를 반환한다.

1. 데스크톱 초기 workspace에서 새 장면을 추가하고 `원고 본문`에 `실패 뒤에도 남을 장면`을 입력한다.
2. 첫 PUT 뒤 `role="alert"`가 `저장 실패`를 포함하고 `button` 이름 `원고 저장 다시 시도`가 나타나는지 확인한다.
3. 실패 상태에서도 `2장 제목 없는 장면`이 current이고 편집기 값이 그대로이며 `새 장면 추가` 버튼은 enabled인지 확인한다.
4. `원고 저장 다시 시도`를 클릭한다.
5. 두 번째 PUT이 최신 두 장면 전체, 두 번째 장면의 입력, 두 번째 장면 `activeSceneId`, `expectedRevision: 1`을 포함하는지 확인한다. `status` 이름 `자동 저장됨`을 기다린다.
6. reload하고 두 번째 장면과 `실패 뒤에도 남을 장면`이 복원되는지 확인한다.

**성공 기준:** 500 응답은 로컬 draft를 지우거나 구조 충돌 dialog를 열지 않으며, 재시도는 동일한 최신 draft를 저장한다.

## 시나리오 6 — 안전한 구조 병합으로 서버 최신 변경과 로컬 새 장면을 모두 유지한다

**추적:** 완료 기준 1, 5, 6; 구조 충돌 해결의 `내 새 장면 유지`.

**네트워크:** 로컬 추가 후 첫 PUT은 409를 반환한다. 이어지는 GET은 revision 7과 서버 최신 원고(초기 1장에 서버 최신 본문, ID `server-only-scene`/제목 `서버에서 추가된 장면`/장 번호 3/본문 `보존해야 하는 서버 변경`인 서버 전용 장면)를 반환한다. 해결 PUT은 성공 revision 8을 반환하고 이후 GET에 반영한다.

1. 초기 workspace에서 새 장면을 추가하고 그 본문에 `병합해 지킬 로컬 새 장면`을 입력한다.
2. 409와 authoritative GET 뒤 `dialog` 이름 `원고 저장 충돌 해결`이 열리고 `서버 최신 원고에 아직 없는 새 장면이 있어요.`가 표시되는지 확인한다. `내 새 장면 유지`, `서버 최신본 적용` 버튼은 enabled이고 diff table은 없어야 하며, 배경의 `새 장면 추가`는 disabled여야 한다.
3. `button` 이름 `내 새 장면 유지`를 클릭한다.
4. 해결 PUT의 `expectedRevision`이 7이고 장면 ID 순서가 `[silver-garden-scene-1, server-only-scene, silver-garden-scene-e2e-local-scene]`인지 확인한다. 서버의 기존/전용 장면 내용과 로컬 새 장면 내용이 각각 보존되고 `activeSceneId`는 로컬 새 장면이어야 한다.
5. 성공 뒤 dialog가 닫히고 `자동 저장됨`, 로컬 새 장면 current, 편집기 `병합해 지킬 로컬 새 장면`을 확인한다.
6. reload 후 세 장면, 같은 순서와 내용, 로컬 새 장면 활성 상태가 복원되는지 확인한다.

**성공 기준:** 최신 서버 revision 위에 로컬 전용 장면만 보수적으로 append하며 어느 쪽 변경도 덮어쓰지 않는다. scene-diffs 호출, 새 엔드포인트 호출, ID/장 번호 충돌을 자동 덮어쓰는 동작은 실패다.

## 시나리오 7 — 서버 최신본 적용으로 로컬 전용 장면과 미저장 내용을 버린다

**추적:** 완료 기준 5, 6; 구조 충돌 해결의 `서버 최신본 적용`.

**네트워크:** 첫 PUT은 409를 반환한다. authoritative GET은 revision 7과 서버 최신 원고(첫 장면 및 ID `server-only-scene`, 제목 `서버에서 추가된 장면`, 장 번호 2, 본문 `서버가 확정한 장면`, 활성 장면 `server-only-scene`)를 반환한다. 선택 후에는 해결 PUT이 없어야 하며 이후 reload GET도 같은 revision 7 원고를 반환한다.

1. 초기 workspace에서 새 장면을 추가하고 `원고 본문`에 `버려질 로컬 전용 본문`을 입력한다.
2. 구조 충돌 dialog와 두 해결 버튼이 준비될 때까지 기다린다.
3. `button` 이름 `서버 최신본 적용`을 클릭한다.
4. dialog가 닫히고 `button` 이름 `2장 서버에서 추가된 장면`이 current, 헤더가 `2장 · 서버에서 추가된 장면`, 편집기 값이 `서버가 확정한 장면`인지 확인한다.
5. `button` 이름 `2장 제목 없는 장면`과 텍스트 `버려질 로컬 전용 본문`이 더 이상 존재하지 않는지 확인한다. 409였던 최초 PUT 외에 추가 PUT이 발생하지 않았는지 확인한다.
6. reload 후에도 서버의 두 장면과 서버 활성 장면만 복원되고 로컬 전용 장면은 돌아오지 않는지 확인한다.

**성공 기준:** 명시적 서버 채택은 전체 서버 원고와 revision을 그대로 채택하고 로컬 전용 구조·내용을 폐기한다. 서버 채택 뒤 로컬 draft를 다시 autosave하거나 로컬 장면이 부활하면 실패다.

## 생성 시 주의사항

- locator는 CSS 구조나 한국어 문구의 부분 정규식보다 위에 명시한 role/name을 우선한다. live region과 `aria-current`만 해당 속성 locator를 사용한다.
- 네트워크 응답 순서는 요청 카운터로 명시적으로 제어하고, 각 PUT request body를 assertion 전에 JSON으로 기록한다.
- reload 검증은 같은 Page에서 실제 `page.reload()`을 사용하되 성공 PUT이 갱신한 테스트 fixture를 GET으로 반환한다. 컴포넌트 재마운트만으로 대체하지 않는다.
- 본 계획 밖의 rename/reorder/delete UI, 충돌 불안전 병합 변형, scene-content 충돌, 새 API operation 테스트를 추가하지 않는다.
