# Muse 종합 웹 테스트 계획

## 개요

실제 탐색한 Muse 앱(`http://127.0.0.1:5173/`)의 작품 서재, 신규 작품 생성, 집필 워크스페이스, 원고 자동 저장, 도메인 패널과 AI 집필 도구를 검증한다.

모든 시나리오는 브라우저 저장소, 캐시, 네트워크 오버라이드와 테스트 데이터를 초기화한 blank/fresh 상태에서 독립적으로 실행한다.

공통 성공 기준:

- 명시된 URL, 텍스트, 접근 가능한 이름, 상태와 데이터가 일치한다.
- mutation은 의도한 횟수만 실행된다.
- 데이터 유실이나 기능 관련 콘솔 오류가 없다.

공통 실패 조건:

- 잘못된 내비게이션
- 중복 생성 또는 저장
- 무한 로딩
- 조용한 데이터 유실
- 접근 불가능한 오류 메시지나 컨트롤

## 1. 작품 서재

### 1.1 서재 기본 표시와 전역 탐색

1. fresh 상태에서 `/`를 열고 프로젝트 조회 완료를 기다린다.
   - 예상: 문서 제목은 `Muse — 로맨스 집필 에이전트`이며 `내 작품`, `설정`, `새 작품 시작`이 표시된다.
2. 프로젝트 카드의 트로프, 제목, 한 줄 아이디어, 최근 집필, `이어 쓰기`를 확인한다.
   - 예상: 카드 데이터와 프로젝트 개수 문구가 API 응답과 일치한다.
3. Tab과 Enter로 헤더와 CTA를 순회한다.
   - 예상: 논리적인 포커스 순서와 보이는 포커스가 제공된다.
   - 실패: 누락·중복 카드, 이름 없는 컨트롤, 기능 관련 콘솔 오류.

### 1.2 기존 작품 집필 재개

1. fresh 상태에서 `은빛 정원의 약속` 카드가 있는 서재를 연다.
   - 예상: 카드는 `/projects/silver-garden/write`를 가리킨다.
2. 카드를 활성화한다.
   - 예상: 해당 집필 URL로 이동하고 프로젝트 제목과 `제1장 · 첫 장면`이 표시된다.
3. 브라우저 뒤로 이동한다.
   - 예상: 서재와 목록 상태가 유지된다.
   - 실패: 다른 프로젝트가 열리거나 목록 상태가 손실된다.

### 1.3 빈 서재와 조회 오류

1. fresh 상태에서 프로젝트 API가 빈 배열을 반환하도록 하고 `/`를 연다.
   - 예상: 영구 로딩 없이 빈 상태와 신규 작품 CTA가 표시되며 가짜 카드는 없다.
2. 별도 fresh 실행에서 API가 4xx, 5xx, 오프라인 응답을 반환하게 한다.
   - 예상: 이해 가능한 한국어 오류와 재시도 수단이 표시된다.
3. 응답을 성공으로 바꾸고 재시도한다.
   - 예상: 목록이 중복 없이 복구된다.
   - 실패: 빈 화면, 무한 재요청, 미처리 예외, 잘못된 프로젝트 개수.

## 2. 신규 작품

### 2.1 트로프 탐색과 선택

1. fresh 상태에서 `/new`를 연다.
   - 예상: `New story · Step 1`, 페이지 제목과 작품 서재 복귀 링크가 표시된다.
2. 다음 트로프 카드를 확인한다.
   - 앙숙에서 연인으로
   - 계약 연애
   - 재회 로맨스
   - 친구에서 연인으로
   - 예상: 각 카드에 고유 설명, 태그 3개, 선택 링크가 있다.
3. 각 카드를 별도 fresh 실행에서 선택한다.
   - 예상: `/new/setup?trope=<id>`로 이동하고 Step 2의 선택 트로프와 기본 문장이 일치한다.
   - 실패: 잘못된 trope ID, 이전 카드의 상태 혼입, 키보드 활성화 실패.

### 2.2 프로젝트 생성 happy path

1. fresh 상태에서 `/new/setup?trope=contract-romance`를 연다.
   - 예상: 선택한 트로프는 `계약 연애`이고 아이디어, `서윤`, `도현` 기본값이 표시된다.
2. 제목에 `테스트 계약 연애`를 입력하고 `작업 공간 열기`를 한 번 활성화한다.
   - 예상: 프로젝트 생성 요청이 정확히 한 번 실행된다.
3. 생성 완료를 기다린다.
   - 예상: `/projects/<id>/write`로 이동하고 제목, 첫 장면, 자동 저장 상태와 원고 본문이 표시된다.
   - 실패: 중복 생성, 입력값 변경, 다중 내비게이션, 잘못된 프로젝트 ID.

### 2.3 폼 경계값·검증·API 오류

1. 각 fresh 실행에서 제목, 아이디어와 인물 이름에 다음 값을 적용한다.
   - 빈값
   - 공백만
   - 1자
   - 허용 최대 길이
   - 최대 길이 + 1
   - 한글, 이모지, 특수문자
   - 예상: 허용 값은 손상 없이 유지되고 금지 또는 초과 값은 필드 가까이에서 차단된다.
2. 동일한 두 주인공 이름, 누락된 trope, 알 수 없는 trope를 시험한다.
   - 예상: 잘못된 프로젝트가 생성되지 않고 트로프 선택 화면 또는 명확한 오류로 안내된다.
3. 생성 API의 4xx, 5xx, timeout 상태에서 제출하고 재시도한다.
   - 예상: 입력값과 선택값이 보존되며 성공 재시도 시 프로젝트가 하나만 생성된다.
   - 실패: 입력 유실, 무한 스피너, 중복 생성, 접근 불가능한 오류.

## 3. 집필 워크스페이스

### 3.1 워크스페이스 구조와 도메인 패널

1. fresh 프로젝트의 `/projects/<id>/write`를 연다.
   - 예상: 서재 복귀, 프로젝트 제목, 현재 장면, 자동 저장 상태, 원고·인물·세계관·AI 도구가 표시된다.
2. `인물 보기`, `세계관 보기`, `원고 보기`를 키보드로 차례로 활성화한다.
   - 예상: 패널과 활성 상태가 정확히 전환되고 작성 중인 원고는 유지된다.
3. `새 장면 추가`와 `1장 첫 장면`을 확인한다.
   - 예상: disabled인 새 장면 추가는 실행되지 않고 첫 장면 선택은 올바른 편집기를 연다.
   - 실패: 패널 중첩, 포커스 손실, 원고 변경, 이름 없는 아이콘 버튼.

### 3.2 원고 편집과 자동 저장

1. fresh 상태에서 원고에 `서윤은 계약서를 내려놓고 도현을 바라보았다.`를 입력한다.
   - 예상: 문장이 한글 손상 없이 편집기에 반영되고 저장 진행 상태가 전달된다.
2. autosave debounce보다 충분히 기다린다.
   - 예상: 적절한 횟수의 저장 요청 후 상태가 `자동 저장됨`으로 돌아온다.
3. 페이지를 새로고침하고 같은 장면을 연다.
   - 예상: 작성 문장이 유지된다.
   - 실패: 저장 요청 폭주, 저장 전 완료 표시, 문자 또는 원고 유실.

### 3.3 저장 실패·충돌·이탈 보호

1. fresh 상태에서 autosave API 네트워크 실패를 만들고 원고를 수정한다.
   - 예상: 저장 실패 상태와 재시도 수단이 표시되고 로컬 내용은 유지된다.
2. 네트워크 복구 후 재시도하고 별도 fresh 실행에서 서버 버전 충돌을 발생시킨다.
   - 예상: 최신 원고가 한 번 저장되고 충돌 시 서버 변경과 내 변경 중 선택하기 전까지 덮어쓰지 않는다.
3. 미저장 상태에서 작품 서재로 복귀한다.
   - 예상: 이탈 경고 또는 안전한 로컬 보존 정책이 적용된다.
   - 실패: 조용한 데이터 유실, 무한 재시도, 거짓 저장 완료 상태.

## 4. AI 집필 도구

### 4.1 AI 패널 상태와 열기·닫기

1. fresh 워크스페이스에서 `AI 도구 열기`를 누른다.
   - 예상: `AI 집필 도구`, 개인정보 안내와 닫기 버튼이 있는 보조 패널이 열린다.
2. 다음 기능 상태를 확인한다.
   - 이어 쓰기
   - 문장 다듬기
   - 대사 제안
   - 일관성 검사
   - 예상: 선택 문장이 없을 때 `문장 다듬기`만 disabled이고 나머지는 사용할 수 있다.
3. 닫기 버튼과 Escape를 각각 독립 실행에서 사용한다.
   - 예상: 패널이 닫히고 포커스가 호출 버튼으로 돌아간다.
   - 실패: 원고 변경, 포커스 함정, 배경 컨트롤의 잘못된 조작.

### 4.2 AI 제안 적용·취소·오류

1. fresh 상태에서 원고를 입력하고 `이어 쓰기`를 요청한다.
   - 예상: 로딩 중 중복 실행이 차단되고 제안은 원고와 구분되어 표시된다.
2. 첫 요청은 취소하고 두 번째 요청은 적용한다.
   - 예상: 취소 시 원고가 변하지 않고 적용 시 제안이 한 번만 정확한 위치에 삽입되어 autosave된다.
3. 각 AI 기능을 4xx, 5xx, timeout, 빈 응답으로 실행한다.
   - 예상: 원고를 변경하지 않고 오류와 재시도 수단을 제공한다.
   - 실패: 자동 덮어쓰기, 영구 로딩, 중복 삽입, 잘못된 응답 적용.

### 4.3 문장 선택 경계와 일관성 검사

1. fresh 상태에서 선택 없음, 공백, 1자, 문장 하나, 여러 문단, 최대 길이, 최대+1을 각각 선택한다.
   - 예상: 유효한 선택에서만 문장 다듬기가 활성화되고 초과 범위는 명확하게 제한된다.
2. 유효한 문장 다듬기 제안을 적용한다.
   - 예상: 선택 범위만 교체되고 주변 원고와 한글 조합이 보존된다.
3. 인물, 장소와 감정선 충돌 응답으로 일관성 검사를 실행한다.
   - 예상: 충돌 근거와 원고 위치가 읽기 쉬운 목록으로 제공된다.
   - 실패: 선택 범위 밖 수정, 선택 유실, 접근 불가능한 결과.

## 5. 접근성·반응형·라우팅

### 5.1 키보드·스크린리더·반응형

1. 각 fresh 실행에서 `/`, `/new`, setup과 workspace의 핵심 흐름을 키보드만으로 완료한다.
   - 예상: landmark, heading, 레이블, 상태와 포커스 순서가 화면 의미와 일치하며 키보드 함정이 없다.
2. 320×568, 768×1024, 1440×900과 200% 확대에서 주요 화면을 연다.
   - 예상: CTA, 폼, 편집기와 패널이 겹치거나 잘리지 않고 불필요한 수평 스크롤 없이 사용할 수 있다.
3. 긴 한국어 제목·이름·아이디어와 선호 감소 모션을 적용한다.
   - 예상: 전체 내용에 접근할 방법과 조작성이 유지된다.
   - 실패: 색상만으로 전달되는 상태, 가려진 포커스, 화면 밖 컨트롤.

### 5.2 직접 URL·404·콘솔 위생

1. fresh 상태에서 알려진 각 경로를 직접 열고 새로고침한다.
   - 예상: SPA fallback으로 동일 화면이 복원되고 서버의 일반 404 문서로 떨어지지 않는다.
2. 알 수 없는 프로젝트 ID와 알 수 없는 경로를 연다.
   - 예상: 안전한 not-found 또는 서재 복귀 상태가 표시되고 다른 프로젝트 데이터가 노출되지 않는다.
3. 핵심 happy path의 콘솔과 네트워크 요청을 확인한다.
   - 예상: 기능 관련 console error, 실패 요청과 중복 mutation이 없다.
   - 참고: 실제 탐색 중 `/favicon.ico` 404 한 건을 관찰했으며 배포 품질 결함으로 기록한다.
   - 실패: favicon 외 기능 리소스 오류, 미처리 예외, 중복 mutation.

## 실행 범위와 제한 사항

실제 탐색으로 확인한 화면:

- `/` 작품 서재
- `/new` 트로프 선택
- `/new/setup?trope=contract-romance`
- 신규 프로젝트 생성 후 `/projects/mock-project-1/write`
- 원고 목차 및 편집기
- AI 집필 도구 패널
- 원고 입력과 autosave 대기
- 기존 작품 카드가 가리키는 `/projects/silver-garden/write`

관찰된 제한 사항:

- backend는 MSW/mock 응답 기준으로 동작했다.
- 인물·세계관 패널의 상세 콘텐츠, 실제 AI 성공/오류 응답과 충돌 대화상자는 계획에 포함했으나 이번 탐색에서 모두 실행하지는 않았다.
- `/favicon.ico` 요청이 404를 반환했다.
